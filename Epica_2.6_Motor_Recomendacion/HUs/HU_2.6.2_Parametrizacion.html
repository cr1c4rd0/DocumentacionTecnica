Parametrización de nuevas variables en el motor de recomendación (sin redeploy)

Épica 2.6 · Motor de Recomendación · Etapa 1 (Desarrollo)

Enunciado

COMO

área de negocio que administra el portafolio y los criterios de recomendación de servicios de la CCB.

QUIERO

definir variables adicionales que influyan en las recomendaciones (nuevas señales, pesos, criterios de ranking), configurar cómo se usan, declararlas en el Feature Store y activarlas sin redeploy del modelo.

PARA

ajustar y enriquecer el motor de recomendación con nuevas dimensiones de negocio de forma ágil, segura y auditada, sin interrupciones de servicio ni intervención del equipo de ingeniería para cada cambio.

1. Criterios de Aceptación E1 – Desarrollo
  1. Definir variables adicionales que puedan influir en el scoring
    • El sistema permite declarar nuevas variables (features) que el motor puede usar en el scoring sin modificar el código del modelo.
    • Ejemplos de nuevas variables: etapa_empresa, meses_como_cliente, indice_actividad, score_potencial_exportador.
    • Cada variable tiene: nombre, tipo (numérico / categórico / binario), fuente de datos, descripción de negocio.
  2. Configurar cómo se usan
    • Para cada variable se define su rol en el scoring: boost, filtro, ponderación, señal de contexto.
    • Los pesos de las variables son configurables mediante archivo YAML/JSON o panel sin redeploy.
    • Posibilidad de activar/desactivar una variable sin eliminarla del sistema.
  3. Declarar variables en el Feature Store
    • Las nuevas variables se declaran formalmente en el Feature Store con su esquema, fuente y periodicidad de actualización.
    • El Feature Store valida la disponibilidad de la variable antes de que el motor la consuma.
    • Si la variable no está disponible para un usuario, el motor usa un valor por defecto configurado.
  4. Activar cambios mediante hot-reload / feature flags
    • Los cambios en variables o pesos se activan en ≤ 5 minutos sin reiniciar el servicio.
    • Mecanismo: config hot-reload (recarga del YAML/JSON) o feature flags.
    • Rollback inmediato a la versión anterior en caso de degradación de métricas.
  5. Documentación y "diccionario de variables"
    • Cada variable registrada en un diccionario de variables accesible para equipo técnico y de negocio.
    • Diccionario incluye: descripción, tipo, fuente, valores válidos, impacto esperado en el scoring, fecha de activación y versión.
    • Generación automática del diccionario desde las declaraciones del Feature Store.
  6. Clasificar variables según su naturaleza operacional
    • Variables de usuario: atributos del perfil (segmento, etapa, actividad).
    • Variables de servicio: atributos del catálogo (categoría, vigencia, popularidad).
    • Variables de contexto: momento, canal, campaña activa.
    • Variables derivadas: calculadas a partir de otras (ej. afinidad = similitud coseno entre embeddings).
  7. Catálogo de variables y editor de reglas de ranking
    • Interfaz (o archivo YAML) que lista todas las variables disponibles con su estado (activa / inactiva) y peso actual.
    • Editor de reglas de ranking: permite ajustar la fórmula de scoring combinando variables con pesos.
2. Tecnología Propuesto E1
Feature Store Ampliable
  • Esquema extensible: nuevas variables se agregan como columnas o entidades sin romper compatibilidad.
  • Validación de esquema al declarar nuevas variables.
  • Valores por defecto para usuarios sin la variable calculada.
Config Hot-Reload
  • YAML de configuración con todos los pesos y variables del motor.
  • El Motor/API recarga la config al detectar cambio (polling o push).
  • Versionado semántico de la config: config-YYYY.MM.DD.
  • Feature flags para activar/desactivar variables individualmente.
Integraciones
  • CRM Dynamics: fuente de variables de usuario (segmento, etapa, actividad).
  • Feature Store: repositorio central de todas las variables del motor.
  • Motor/API: consume variables en tiempo de inferencia.
  • Frontend/UI: catálogo visual de variables y editor de pesos.
  • Formulario: interfaz de declaración de nuevas variables para el equipo de negocio.
Auditoría y Rollback
  • Cada cambio de variable/peso auditado con: usuario, timestamp, versión anterior, versión nueva.
  • Rollback en un click a cualquier versión anterior del config.
  • Alerta automática si las métricas del motor degradan tras un cambio.
3. Flujo Técnico y de Datos E1 – Desarrollo
Pasos del Pipeline de Parametrización
  1. Definir variables adicionales: equipo de negocio declara nueva variable en CRM Dynamics (nombre, tipo, fuente, descripción).
  2. Configurar cómo se usan: Feature Store registra la variable con su rol (boost/filtro/ponderación) y valor por defecto.
  3. Declarar: Motor/API incorpora la variable en la fórmula de scoring via config YAML.
  4. Activar cambios: Frontend/UI aplica el cambio vía hot-reload; Formulario confirma activación al equipo.
  5. Proveer diccionario de variables: CRM persiste la documentación actualizada con todos los metadatos de la variable.
  6. Clasificar variables: CRM Dynamics categoriza la variable (usuario / servicio / contexto / derivada).
  7. Catálogo de variables: Feature Store muestra el estado de todas las variables activas con sus pesos.
  8. Editor de reglas de ranking: Motor/API expone el editor donde se ajusta la fórmula de scoring.
Ejemplo de Declaración de Variable en YAML
variables:
  - id: "VAR-018"
    nombre: "score_potencial_exportador"
    tipo: "numerico"
    rango: [0.0, 1.0]
    fuente: "CRM_Dynamics"
    actualizacion: "semanal"
    rol: "boost"
    peso: 0.15
    valor_default: 0.0
    activa: true
    descripcion: "Puntuación de potencial exportador calculada por el área comercial."
    clasificacion: "variable_usuario"
    auditoria:
      creado_por: "mrodriguez@ccb.org.co"
      timestamp: "2025-03-01T14:00:00Z"
      version_config: "config-2025.03.01"
4. Diseño — Catálogo de Variables y Editor de Ranking Admin interno
Catálogo de Variables
  • Tabla con: ID, nombre, tipo, clasificación, fuente, peso actual, estado (activa/inactiva).
  • Filtros por clasificación (usuario / servicio / contexto / derivada).
  • Acciones: activar/desactivar, editar peso, ver historial de cambios.
  • Enlace a la documentación (diccionario) de cada variable.
Editor de Reglas de Ranking
  • Fórmula de scoring visualizada con pesos editables por variable.
  • Sliders de peso con validación de suma total = 1.0.
  • Botón "Aplicar" con hot-reload (sin redeploy).
  • Vista de impacto: cambio en top-5 para una muestra de usuarios de prueba.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Nueva variable declarada Variable declarada en YAML aparece en el catálogo con estado "inactiva" hasta activación explícita.
2 Activación sin redeploy Variable activada en ≤ 5 min; log registra nuevo version_config; motor la usa en inferencias siguientes.
3 Cambio de peso → cambio en ranking Aumentar el peso de score_potencial_exportador de 0.1 a 0.3 produce cambio verificable en el top-5 de usuarios exportadores.
4 Valor por defecto cuando falta la variable Usuario sin score_potencial_exportador calculado recibe 0.0 como valor; motor no falla.
5 Rollback Tras degradación de métricas, rollback a versión anterior del config en ≤ 2 minutos; motor restaura comportamiento previo.
6 Diccionario de variables actualizado Cada nueva variable declarada aparece en el diccionario con todos sus metadatos; generación automática.
6. Glosario
Feature / Variable
Atributo o señal que el motor de recomendación usa como entrada para calcular el score de un servicio para un usuario.
Feature Store
Repositorio centralizado de variables/features de usuario y servicio, precalculadas y listas para consumo en tiempo de inferencia.
Diccionario de Variables
Catálogo documentado de todas las variables del motor, con descripción de negocio, tipo, fuente, impacto y estado.
Config Hot-Reload
Mecanismo que recarga la configuración del motor (pesos, variables activas) sin reiniciar el servicio.
Feature Flag
Switch que permite activar o desactivar una variable o funcionalidad en tiempo de ejecución sin redeploy.
Editor de Ranking
Interfaz que permite ajustar la fórmula de scoring del motor modificando los pesos de las variables de forma visual.
Nota de alcance E1: El catálogo y editor pueden implementarse como archivos YAML en repositorio con PR review. Lo esencial es que las variables se puedan declarar, activar y desactivar sin redeploy, con diccionario documentado y auditoría de cambios. La UI visual se desarrolla en etapas posteriores.
