Épica 2.2 · Panel de Administración y Explicabilidad · XAI para Negocio

Enunciado

COMO

administrador/a de negocio de la CCB.

QUIERO

ver la lógica explicativa detrás de cada recomendación (qué factores la soportan y qué reglas la afectaron).

PARA

validar pertinencia, auditar decisiones, ajustar reglas y comunicar con confianza a las áreas interesadas.

1. Criterios de Aceptación Definition of Done
  1. Explicación clara por recomendación (XAI legible)
    • En la vista de una recomendación, el panel muestra Top-3 factores que más aportaron al score (p. ej.: "Sector comercio +0.18", "Usó Certificados hace 14 días +0.12", "Servicios gratuitos priorizados +0.07").
    • Cada factor incluye tipo (usuario/empresa, servicio, comportamiento, regla) y un resumen humano ("Por tu sector y tu uso reciente de certificados.").
  2. Trazabilidad completa (end-to-end)
    • Se visualizan eventos y decisiones: generación de candidatos → filtros (listas blancas/negras, cupos/elegibilidad) → re-ranking → boosts/pinning → Top-N final.
    • Se puede abrir un log de decisiones con timestamp, score antes/después de cada etapa y regla aplicada.
  3. Compatibilidad con usuarios anónimos (consentimiento)
    • Si el usuario es anónimo con anon_id, la explicación nunca expone PII.
    • Se indica el estado de consentimiento ("Personalización: ON/OFF").
  4. Rendimiento y alcance
    • Carga de explicación ≤ 1.0 s (p95) para un ítem con datos cacheados.
    • Existe búsqueda por usuario (anon_id o id CRM), segmento, servicio y fecha.
  5. Exactitud y consistencia
    • La suma de contribuciones mostradas reconstruye el score (±2% tolerancia numérica).
    • Si una regla invalida un servicio (lista negra/cupo agotado), ese servicio no aparece; queda motivo visible.
  6. Gobernanza y exportación
    • Botón "Exportar evidencia" (PDF/CSV) con: factores, reglas activas y métricas del caso (CTR/CR, impresiones).
    • Bitácora: quién consultó/descargó qué explicación (auditoría).
  7. Accesibilidad y lenguaje
    • Panel WCAG 2.1 AA; idioma ES/EN.
    • Todos los términos técnicos tienen tooltip con glosario breve.
2. Tecnología Propuesto E1
XAI — Explainable AI
  • Exposición de contribuciones de variables (features) al score.
  • En re-ranking (Learning-to-Rank): SHAP o feature contributions equivalentes (para LR/XGBoost/LightGBM).
Motor de Reglas & Decision Logging
  • Capa que aplica listas blancas/negras, boost gratuitos, pinning, filtros de elegibilidad/cupos.
  • Registro estructurado de cada paso del pipeline (candidatos, filtros, re-rank, score final) con IDs correlacionados.
Feature Store & API de Consulta
  • Variables efectivamente utilizadas (con versión y timestamp) para reproducibilidad.
  • GET /explanations/{userId|anon_id}/{service_id}?at=timestamp → factores, reglas y log.
  • GET /rules/active → reglas vigentes.
  • GET /recs?userId|segment → Top-N con explicación embebida (opcional).
Capa de Presentación
  • Panel web con atajos de teclado, labels ARIA, contraste AA.
  • Tooltips con glosario para términos como "SHAP", "pinning", "boost".
  • Acción "Abrir en sandbox" para simular impacto de cambios de pesos.
3. Flujo Técnico XAI Alto Nivel
Pipeline de Recomendación con Trazabilidad
  1. Generación de candidatos (Stage 1): Collaborative Filtering + Content-Based → Top-K inicial. Se excluyen ya tomados/descartados y se aplican filtros de elegibilidad/cupos.
  2. Re-ranking (Stage 2): modelo L2R calcula score base; se computan contribuciones por feature (SHAP/coeficientes); se aplican reglas (boost gratuitos, pinning) registrando los deltas al score.
  3. Salida + logging: se arma el Top-N final; se guarda el decision log: candidatos, scores por etapa, reglas aplicadas, Top-3 factores, IDs de features/versión de modelo.
  4. Consulta en el panel: el panel llama a la API de explicaciones con userId/anon_id + service_id (y timestamp). La API rehidrata: factores (con texto legible), reglas activas, eventos, métricas y consentimiento.

4. Diseño — Vista "Detalle de Recomendación" en el Panel
Tarjeta "Score y Razones"
  • Header: Servicio · Usuario (anon_id o id CRM) · Fecha/hora · Modelo vX.Y · Reglas activas (chips).
  • Score final con mini-barras: score base + boosts reglas − penalizaciones.
  • Top-3 razones (texto humano):
    • "Eres microempresa del sector comercio (+0.18)"
    • "Has usado Certificados en 14 días (+0.12)"
    • "Regla Gratuitos primero (+0.07)"
  • Switch "Ver detalle técnico" → lista de contribuciones por feature (tabla: variable, valor, contribución).
Sección "Trazabilidad" (timeline)
  • Candidatos (K=200) → Filtros (−12) → Re-ranking (N=12) → Boost/Pin → Top-N final.
  • Cada etapa expandible muestra IDs y cambios de score.
  • Panel lateral "Reglas y políticas": reglas vigentes (listas, boosts, pinning, cupos) con motivo y vigencia.
  • Pie: CTR/CR históricos, impresiones, ocultados, dwell medio.
  • Acciones: Exportar evidencia (PDF/CSV), Abrir en sandbox.
5. Escenarios de Prueba Gherkin
# Escenario Resultado Esperado

6. Glosario
XAI (Explainable AI)
Técnicas para explicar por qué el modelo decide algo de forma comprensible para humanos.
SHAP
Método que estima la contribución de cada feature al score del modelo (Shapley Additive Explanations).
Feature
Variable del modelo (p. ej.: "sector=CIIU G", "gratuito=true", "días desde último uso=14").
Re-ranking / L2R
Modelo que ordena candidatos con más señales (Learning-to-Rank).
Boost
Aumento de score por regla de negocio (p. ej.: "gratuitos primero").
Pinning
Fijar un ítem en posiciones superiores por campaña/política de negocio.
Decision Log
Registro paso a paso de cómo se llegó al Top-N, incluyendo scores, reglas y deltas por etapa.
CTR / CR
Tasas de clic (Click-Through Rate) y de conversión (Conversion Rate).
Parámetros iniciales (ajustables en E1): Top-3 razones como estándar (botón "ver más" para lista completa) · Latencia objetivo: explicación ≤ 1 s (p95) usando cache de contribuciones · Privacidad: nunca mostrar valores crudos de PII · Versionado: toda explicación referencia modelo vX.Y, feature set vA.B y reglas (hash/fecha).
