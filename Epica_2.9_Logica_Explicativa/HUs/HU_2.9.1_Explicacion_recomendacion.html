Explicación por recomendación

Épica 2.9 · Lógica Explicativa · Panel de auditoría y trazabilidad por ítem

Enunciado

COMO

administrador del Recomendador de Servicios CCB (o analista autorizado).

QUIERO

consultar una explicación detallada por cada recomendación individual: qué variables, reglas y fuentes influyeron en la decisión, con trazabilidad completa de IDs de modelo y versión.

PARA

auditar las decisiones del motor, detectar sesgos, validar que las reglas de negocio se aplican correctamente y generar confianza en el sistema entre los equipos de negocio y tecnología.

1. Criterios de Aceptación Explicabilidad
  1. Header Global CCB y acceso al panel
    • El administrador accede al Panel de Administración del Recomendador desde el Header Global CCB con autenticación de rol admin o analyst.
    • El panel está disponible en la intranet CCB; no expuesto al usuario final.
  2. Sidebar de Navegación del Panel
    • El panel incluye un sidebar de navegación con secciones: Recomendaciones, Explicaciones, Reportes, Configuración.
    • La sección "Explicaciones" es accesible directamente desde el sidebar.
  3. Panel de Filtros para búsqueda de explicaciones
    • El panel de filtros permite buscar explicaciones por:
      • user_key_pseudo (identificador seudonimizado del usuario).
      • service_id (servicio recomendado).
      • Rango de fechas (from_date / to_date).
      • Versión del modelo (version_modelo).
      • Tipo de motor: Reglas, ML, Híbrido.
    • Los filtros son combinables; los resultados se actualizan al aplicar.
  4. Grid de Resultados
    • Los resultados de búsqueda se presentan en un grid/tabla con columnas: recommendation_id, user_key_pseudo, service_id, nombre, score, tipo_motor, timestamp.
    • El grid es paginado (máx. 50 registros por página) y ordenable por columna.
    • Latencia de consulta del grid ≤ 500 ms.
  5. Modal de Explicación — botón "Ver explicación"
    • Cada fila del grid expone un botón "Ver explicación" que abre un modal o panel lateral con el detalle completo:
      • Variables influyentes: lista de features con peso relativo (%) ordenada de mayor a menor impacto.
      • Reglas aplicadas: lista de reglas de negocio que se ejecutaron (nombre, resultado, efecto en score).
      • Fuentes de datos: qué sistemas contribuyeron (CRM, SIREP, Catálogo).
      • Tipo de motor: Reglas / ML / Híbrido.
      • Trazabilidad: recommendation_id, version_modelo, version_config, timestamp, latencia_ms.
      • Razones legibles: mismas razones[] que recibió el usuario, en lenguaje natural.
    • El modal permite exportar la explicación individual en JSON.
    • Latencia de carga del modal ≤ 500 ms.
  6. Privacidad y seguridad
    • Solo se muestran identificadores seudonimizados (user_key_pseudo); nunca PII en el panel.
    • Las explicaciones se retienen por un período configurable (defecto: 90 días).
    • El acceso al panel queda auditado en log de accesos.
2. Tecnología Stack Propuesto
Motor de Explicabilidad
  • SHAP (SHapley Additive exPlanations) para modelos ML: calcula contribuciones por feature.
  • LIME como alternativa para modelos de caja negra.
  • Para motores basados en reglas: traza directamente qué condiciones se dispararon y su efecto en el score.
  • Motor desacoplado del pipeline principal: genera y almacena explicaciones de forma asíncrona.
Almacenamiento de Explicaciones
  • Store separado (tabla / colección NoSQL) indexado por recommendation_id.
  • Retención configurable con TTL (defecto: 90 días).
  • Particionado por fecha para consultas eficientes en rangos temporales.
API de Explicaciones
  • GET /explanations?filters... — lista de explicaciones con paginación.
  • GET /explanations/{recommendation_id} — detalle completo de una recomendación.
  • GET /explanations/{recommendation_id}/export — descarga JSON de la explicación.
  • Latencia objetivo: ≤ 500 ms para detalle individual.
Panel Admin — Frontend
  • SPA interna (React / Vue) con sidebar de navegación y panel de filtros.
  • Grid de resultados con paginación server-side.
  • Modal de detalle con renderizado de features en gráfico de barras horizontal (peso relativo).
  • Botón de exportación JSON por recomendación.
3. Flujo Técnico y de Datos Secuencia Panel Admin
ACTORES

Usuario / Actor (Admin) · CRM / Dynamics · Motor / API · Panel Admin · Frontend / UI · Formulario

DISPARADOR

El administrador necesita auditar una recomendación específica: identifica al usuario (pseudo), el servicio y el momento en que se generó la recomendación.

RESULTADO

Explicación completa de la recomendación auditada: features, reglas, fuentes, trazabilidad y razones legibles. Exportable en JSON.

Pasos del Flujo (según Diagramas)
  1. Header Global CCB: el Admin se autentica en el portal CCB; el CRM / Dynamics valida rol y permisos.
  2. Sidebar de Navegación: el Admin selecciona "Explicaciones" en el sidebar; el Motor / API carga el contexto del panel.
  3. Panel de Filtros: el Admin configura los filtros de búsqueda; el Panel Admin envía la consulta al Frontend / UI.
  4. Grid de Resultados: el Frontend / UI llama a GET /explanations?filters; el Formulario renderiza el grid paginado.
  5. Modal de Explicación: el Admin hace clic en "Ver explicación"; el sistema llama a GET /explanations/{id} y renderiza el modal con features, reglas y trazabilidad. El Admin puede exportar el JSON.

4. Diseño — Panel de Administración
Grid de Explicaciones
  • Tabla con columnas: ID recomendación, Usuario (pseudo), Servicio, Score, Tipo Motor, Fecha.
  • Fila expandible o botón "Ver explicación" por registro.
  • Paginación (50/página), orden por columna, búsqueda global en la tabla.
  • Exportación del grid completo en CSV.
Modal de Detalle
  • Gráfico de barras horizontal: features con peso relativo (%).
  • Lista de reglas aplicadas con efecto en score (badge verde/naranja).
  • Sección de trazabilidad: IDs, versiones, timestamp, latencia.
  • Razones legibles (las mismas que recibió el usuario) en caja destacada.
  • Botón "Exportar JSON" en la parte inferior del modal.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Búsqueda por user_key_pseudo El grid muestra todas las recomendaciones del usuario filtrado; latencia ≤ 500 ms.
2 Ver explicación de un ítem El modal se abre con features, reglas, fuentes y trazabilidad completa; latencia ≤ 500 ms.
3 Exportar explicación en JSON Se descarga un archivo JSON válido con todos los campos de la explicación.
4 Filtros combinados Filtrar por version_modelo + rango de fechas devuelve solo registros que cumplen ambos criterios.
5 Distinción tipo motor El modal indica claramente si la recomendación fue generada por Reglas, ML o Híbrido.
6 Sin PII En ninguna vista del panel aparecen nombres, emails u otros datos personales; solo user_key_pseudo.
7 Expiración de explicación Una explicación con antigüedad mayor al TTL configurado no aparece en resultados.
6. Glosario
SHAP
SHapley Additive exPlanations — método matemáticamente riguroso para asignar contribuciones de cada feature a la predicción del modelo.
LIME
Local Interpretable Model-agnostic Explanations — método que aproxima localmente el comportamiento de un modelo complejo para explicar predicciones individuales.
recommendation_id
Identificador único de cada recomendación generada, usado para vincular la predicción con su explicación almacenada.
Peso relativo
Proporción (%) en que cada feature contribuyó al score final de la recomendación.
TTL
Time-to-Live — período máximo de retención de las explicaciones en el store antes de ser purgadas automáticamente.
user_key_pseudo
Identificador seudonimizado del usuario, no reversible, usado en el panel para preservar la privacidad de datos personales.
Nota de alcance: El motor de explicabilidad opera de forma asíncrona y desacoplada del pipeline principal de recomendaciones. Las explicaciones se almacenan en un store separado indexado por recommendation_id. El panel es exclusivo para administradores y analistas; no expone PII en ninguna vista. La latencia de consulta objetivo es ≤ 500 ms por recomendación individual.
