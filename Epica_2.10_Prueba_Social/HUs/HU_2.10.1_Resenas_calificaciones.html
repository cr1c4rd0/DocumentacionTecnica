Historia de Usuario · HU 2.10.1

Módulo de reseñas y calificaciones

Épica 2.10 · Prueba Social · Etapa 1 (Desarrollo)

Enunciado

COMO

usuario empresario que consulta el Recomendador de Servicios.

QUIERO

ver el rating promedio y las reseñas de otros empresarios para cada servicio recomendado.

PARA

tomar decisiones más informadas basadas en experiencias reales de la comunidad empresarial.

1. Criterios de Aceptación E1 – Desarrollo
  1. Visualización por servicio (rating + reseñas)
    • Cada tarjeta de servicio recomendado muestra: rating promedio (estrella + valor numérico, p. ej. ★ 4.2) y número de reseñas como indicador de confiabilidad.
    • Listado de reseñas con paginación lazy load (10 reseñas por página); cada reseña muestra: rating individual, texto, fecha y perfil básico del autor (sin exponer PII completo).
    • Módulo carga de forma asíncrona para no bloquear la visualización de las recomendaciones principales.
  2. Cálculo correcto y trazable del rating
    • El rating promedio se calcula como media ponderada de todas las calificaciones aprobadas para el servicio.
    • El cálculo es reproducible y auditable: se puede trazar qué calificaciones componen el promedio actual.
    • Se registra en log: service_id, rating_calculado, num_resenas, timestamp_calculo, version_algoritmo.
    • Cambios en el algoritmo de cálculo quedan versionados y solo aplican hacia adelante (sin alterar histórico).
  3. Rendimiento
    • Tiempo de carga del módulo de reseñas: ≤ 1 segundo (p95) desde que el usuario solicita ver las reseñas de un servicio.
    • Cache de ratings con TTL configurable (valor de referencia: 1 hora); se invalida automáticamente al publicarse una nueva reseña aprobada.
    • Imágenes de perfil de autores servidas vía CDN; tamaño optimizado.
  4. Moderación y seguridad (contenido)
    • Solo se muestran reseñas con estado aprobada; las reseñas en estado pendiente o rechazada nunca son visibles para otros usuarios.
    • El contenido de las reseñas no expone PII (email, teléfono, NIT); si se detecta PII en texto libre, la reseña va a revisión humana (ver HU 2.10.2).
    • La API de reseñas es independiente del motor de recomendación; un fallo en reseñas no interrumpe las recomendaciones.
  5. Fallback por ausencia de reseñas
    • Cuando un servicio no tiene reseñas aprobadas, se muestra un mensaje elegante: "Sé el primero en calificar este servicio" con CTA para dejar reseña.
    • Si el módulo de reseñas no responde en tiempo, se muestra el bloque de servicio sin el módulo de reseñas (degradación grácil).
  6. Eventos a registrar
    • review_module_viewed: usuario abre la sección de reseñas de un servicio.
    • review_rating_calculated: se recalcula el promedio de un servicio.
    • review_page_loaded: se carga una página de reseñas (con número de página).
    • review_fallback_shown: se muestra el mensaje de "sin reseñas".
    • Todos los eventos incluyen: service_id, user_key_pseudo, timestamp, canal.
2. Tecnología Propuesto E1
API de Reseñas (independiente)
  • Microservicio REST desacoplado del motor de recomendación.
  • Endpoints: GET /services/{id}/reviews, GET /services/{id}/rating, POST /reviews.
  • Paginación con cursor para lazy load eficiente.
Almacenamiento y Cache
  • Base de datos relacional para reseñas (PostgreSQL o similar) con índice por service_id y estado.
  • Cache de ratings en Redis con TTL configurable (default 1h).
  • CDN para imágenes de perfil de autores.
Cálculo de Rating
  • Media ponderada sobre calificaciones con estado aprobada.
  • Versionado del algoritmo de cálculo en tabla de configuración.
  • Job batch para recalcular promedios tras moderación masiva.
Observabilidad
  • Logs estructurados (JSON) por cada cálculo de rating y carga de reseñas.
  • Métricas: latencia p95 del módulo, tasa de cache hit, tasa de fallback activado.
  • Alertas por degradación del SLA de carga (> 1s p95).
3. Flujo Técnico y de Datos E1 – Desarrollo
ACTOR

Usuario empresario que consulta el Recomendador de Servicios y solicita ver el detalle de reseñas de un servicio recomendado.

PARTICIPANTES
  • Usuario / Actor
  • Motor / API (recomendador + API de reseñas)
  • Base de Datos (reseñas + cache)
SALIDA ESPERADA

Rating promedio + lista paginada de reseñas aprobadas, o mensaje de fallback elegante si no hay reseñas disponibles.

Pasos del Flujo (según Diagrama de Secuencia)
  1. Visualización por servicio (rating + reseñas): el usuario solicita al Motor/API la información de rating y reseñas para un servicio específico.
  2. Cálculo correcto y trazable del rating: la Base de Datos devuelve las calificaciones aprobadas; el Motor calcula el promedio ponderado y lo versiona en log.
  3. Rendimiento: el Motor consulta el cache (Redis); si hay hit, devuelve desde cache; si no, recalcula y almacena con TTL.
  4. Moderación y seguridad (contenido): la Base de Datos filtra solo reseñas con estado aprobada; el Motor verifica ausencia de PII antes de serializar la respuesta.
  5. Fallback por ausencia de reseñas: si no existen reseñas aprobadas para el servicio, el Motor devuelve estructura de fallback con mensaje y CTA.
  6. Eventos a registrar: se emite el evento analítico correspondiente (review_module_viewed, review_fallback_shown, etc.) y se devuelve la respuesta final al usuario.

4. Diseño — Qué ve el Usuario Componente de tarjeta de servicio
Tarjeta de Servicio (rating visible)
  • Estrella + valor numérico: ★ 4.2 (47 reseñas).
  • Enlace/botón "Ver reseñas" que despliega el panel lazy.
  • Badges de contexto: Verificado, Nuevo (si aplica).
Panel de Reseñas (lazy load)
  • Lista de reseñas con rating, texto, fecha y perfil básico del autor.
  • Botón "Ver más reseñas" al llegar al final de la página actual.
  • Fallback elegante: "Sé el primero en calificar este servicio" cuando no hay reseñas.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Rating con reseñas aprobadas Se muestra rating promedio correcto (media ponderada de calificaciones aprobadas) y número de reseñas.
2 Servicio sin reseñas Se muestra fallback elegante; no se expone rating de 0 ni campo vacío.
3 Reseña pendiente/rechazada Las reseñas no aprobadas no aparecen en el listado ni afectan el promedio visible.
4 Lazy load paginado Se cargan 10 reseñas iniciales; al paginado siguiente se cargan 10 más sin recargar la página.
5 Rendimiento ≤ 1s El módulo de reseñas responde en p95 ≤ 1000ms; fallo del módulo no bloquea las recomendaciones.
6 Cache hit / invalidación Tras publicar una nueva reseña aprobada, el rating en cache se invalida y el siguiente request devuelve valor actualizado.
7 Trazabilidad de eventos Cada visualización del módulo genera evento review_module_viewed con campos obligatorios.
6. Glosario
Rating promedio
Media ponderada de todas las calificaciones aprobadas para un servicio, expresada en escala 1–5.
Lazy load
Técnica de carga diferida: el contenido se solicita solo cuando el usuario lo necesita (scroll o clic), evitando carga innecesaria.
TTL (Time To Live)
Tiempo de vida de un dato en cache; tras expirar, el dato se recalcula desde la fuente.
Estado de reseña
Ciclo de vida: pendiente → aprobada / rechazada. Solo las aprobadas son públicas.
Fallback elegante
Comportamiento de degradación grácil: el sistema muestra un estado alternativo informativo en vez de un error o campo vacío.
CDN
Content Delivery Network: red de distribución que sirve activos estáticos (imágenes) desde nodos geográficamente cercanos al usuario.
Nota de alcance E1: El módulo de reseñas opera de forma asíncrona e independiente del motor de recomendación. Un fallo en reseñas no interrumpe la experiencia principal de recomendación. La moderación del contenido de las reseñas se especifica en HU 2.10.2.
