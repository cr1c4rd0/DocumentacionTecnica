Navegación integrada con asistente

Épica 2.12 · Micro-frontend Recomendador · Etapa 1 (Desarrollo)

Enunciado

COMO

usuario del portal (empresario) que interactúa con el asistente conversacional y el Recomendador.

QUIERO

moverme fluidamente entre el asistente y el recomendador manteniendo mi contexto de sesión y conversación.

PARA

continuar mi exploración sin interrupciones ni necesidad de reautenticarme o repetir información.

1. Criterios de Aceptación E1 – Desarrollo
  1. Sesiones enlazadas y contexto persistente
    • La sesión del usuario es compartida entre el asistente conversacional y el recomendador: un único token de sesión válido para ambos.
    • El contexto de la conversación (intenciones identificadas, servicios mencionados, preferencias expresadas) se transfiere al recomendador al navegar desde el asistente.
    • El recomendador puede recibir el contexto de conversación como parámetro de entrada para personalizar las recomendaciones mostradas.
  2. Traspaso de "handoff" con evidencia (deep link + payload)
    • Cuando el asistente recomienda que el usuario explore el recomendador, genera un deep link con payload de contexto (JWT corto): intención detectada, entidades mencionadas, historial resumido.
    • El recomendador usa el payload del handoff para presentar recomendaciones relevantes al contexto de la conversación previa.
    • La transición visual entre asistente y recomendador ocurre en ≤ 300 ms (animación de transición fluida).
  3. Señales derivadas de la navegación
    • Las acciones del usuario en el recomendador (servicios vistos, CTA clicados, tiempo en tarjeta) se retroalimentan al asistente como señales de comportamiento.
    • El asistente puede usar estas señales para refinar sugerencias en la conversación posterior.
    • Las señales se transmiten de forma asíncrona y no bloquean la navegación del usuario.
  4. Rendimiento y consistencia UX
    • La transición asistente → recomendador preserva el estado visual del portal (menú, encabezado, sesión activa visible).
    • El regreso al asistente (botón "Volver al asistente") preserva el hilo de conversación exactamente donde se dejó.
    • No existe "doble login" entre el asistente y el recomendador.
    • Tiempo de transición total (asistente → recomendador con contexto cargado): ≤ 300 ms.
  5. Seguridad: no exponer datos sensibles en URL
    • El contexto de conversación no se expone en la URL de forma legible; viaja en un JWT firmado de vida corta (TTL ≤ 10 minutos) o en el body de un PostMessage.
    • No se incluye PII del usuario en el payload del handoff; solo user_key_pseudo e intenciones genéricas.
    • El token de handoff es de uso único: se invalida tras ser consumido por el recomendador.
  6. Fallback cuando no se puede recuperar el contexto
    • Si el payload de contexto del handoff no puede desencriptarse o está expirado, el recomendador carga de forma genérica (sin contexto de conversación) sin mostrar error al usuario.
    • Si el asistente no está disponible al intentar regresar, el usuario permanece en el recomendador con un mensaje informativo.
    • Se registra en log el motivo del fallback: token_expirado, asistente_no_disponible, contexto_invalido.
  7. Prevención de loops exploratorios
    • El sistema detecta si el usuario lleva más de N transiciones (configurable, default 5) entre asistente y recomendador en la misma sesión sin completar ninguna acción.
    • Si se detecta el patrón, el asistente ofrece proactivamente un resumen y una acción directa (p. ej., "¿Quieres inscribirte directamente en estos 3 servicios?").
    • La detección de loops no interrumpe la navegación; es una sugerencia, no un bloqueo.
2. Tecnología Propuesto E1
Sesión Compartida
  • Shared session store (Redis) accesible tanto por el asistente como por el recomendador.
  • JWT con claims de contexto de conversación: intent, entities, session_id, user_key_pseudo.
  • BroadcastChannel API para sincronización de estado entre tabs del mismo origen.
Handoff Asistente → Recomendador
  • PostMessage API para traspaso de contexto cuando ambos están en el mismo documento.
  • JWT de handoff (uso único, TTL 10 min) cuando la navegación implica cambio de URL.
  • El recomendador consume el token de handoff en el primer request y lo invalida.
Señales de Navegación
  • Custom Events del componente recomendador (ccb-rec-view, ccb-rec-cta-click) capturados por el asistente.
  • Pipeline asíncrono: señales encoladas y procesadas sin bloquear la UI.
  • Motor de Recomendación consume señales para actualizar el perfil de sesión.
Detección de Loops
  • Contador de transiciones asistente ↔ recomendador en sesión (almacenado en Redis).
  • Umbral configurable (default N=5) en tabla de config; hot-reload.
  • Al detectar loop: el asistente emite sugerencia de acción directa, no un bloqueo.
3. Flujo Técnico y de Datos E1 – Desarrollo
ACTOR

Usuario del portal (empresario) que alterna entre el asistente conversacional y el recomendador.

PARTICIPANTES
  • Usuario / Actor
  • Motor / API (recomendador + gestión de sesión)
  • Frontend / UI (portal + componente recomendador)
  • Asistente / Chat (sistema conversacional)
CONTRATO DE HANDOFF
// JWT Payload (handoff)
{
  "session_id": "sess-42",
  "user_key_pseudo": "u_9f3a",
  "intent": "formalizacion",
  "entities": ["MiPyme", "Bogotá"],
  "use_once": true,
  "exp": 1703163900
}
Pasos del Flujo (según Diagrama de Secuencia)
  1. Sesiones enlazadas y contexto persistente: el Usuario interactúa con el asistente; el Motor/API crea y comparte la sesión en Redis con el recomendador desde el inicio.
  2. Traspaso de handoff: el Asistente/Chat genera el JWT de handoff con contexto de conversación y lo pasa al Frontend/UI para la transición al recomendador.
  3. Señales derivadas de la navegación: el Asistente/Chat recibe señales de comportamiento del usuario en el recomendador (servicios vistos, clics) y las envía al Motor/API para retroalimentación.
  4. Rendimiento y consistencia UX: el Motor/API actualiza el perfil de sesión con las señales y devuelve recomendaciones actualizadas al Frontend/UI en ≤ 300ms.
  5. Seguridad — no exponer datos en URL: el Frontend/UI valida que el contexto no viaja en query string; usa PostMessage o JWT de uso único; el Asistente/Chat confirma token consumido.
  6. Fallback cuando no se puede recuperar contexto: el Asistente/Chat detecta contexto inválido o expirado; el Motor/API devuelve respuesta de fallback (carga genérica del recomendador).
  7. Prevención de loops exploratorios: el Motor/API detecta el patrón de N transiciones sin acción; el sistema devuelve sugerencia de acción directa al usuario.

4. Diseño — Qué ve el Usuario Transición asistente ↔ recomendador
Desde el Asistente
  • Botón "Ver en recomendador" en las respuestas del asistente que incluyen sugerencias de servicios.
  • Transición visual fluida (fade/slide ≤ 300ms) hacia el recomendador.
  • Indicador visual de que el contexto de conversación se transfirió.
Desde el Recomendador
  • Botón "Volver al asistente" visible en la interfaz del recomendador.
  • Al volver, el hilo de conversación se restaura en el punto exacto donde se dejó.
  • Indicador de sesión activa compartida (sin doble login).
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Transición asistente → recomendador Recomendador carga en ≤ 300ms con recomendaciones contextualizadas por la intención del asistente.
2 Regreso al asistente Hilo de conversación restaurado exactamente en el punto donde se dejó; sin re-login.
3 Token de handoff expirado Recomendador carga de forma genérica (sin contexto); log registra token_expirado; sin error visible al usuario.
4 Token de uso único (reintento) Segundo intento de usar el mismo token resulta en rechazo silencioso; recomendador carga genéricamente.
5 Señales de navegación retroalimentadas Clic en servicio desde el recomendador es capturado por el asistente como señal; próxima respuesta del asistente refleja el interés.
6 Detección de loop (N=5 transiciones) Al alcanzar el umbral, el asistente presenta sugerencia de acción directa; la navegación no se bloquea.
7 Asistente no disponible al volver El usuario permanece en el recomendador con mensaje informativo; log registra asistente_no_disponible.
6. Glosario
Handoff
Traspaso de contexto y control entre dos sistemas (asistente y recomendador); incluye la evidencia del contexto de conversación previa.
Sesión compartida
Sesión de usuario única válida para ambos sistemas (asistente y recomendador), almacenada en Redis y referenciada por session_id.
BroadcastChannel
API del navegador para comunicación entre pestañas/ventanas del mismo origen sin necesidad de servidor.
Token de uso único
Token que se invalida automáticamente tras ser consumido por primera vez, previniendo su reutilización o replay.
Loop exploratorio
Patrón de comportamiento en el que el usuario alterna repetidamente entre el asistente y el recomendador sin completar ninguna acción concreta.
Señal de navegación
Evento de comportamiento del usuario en el recomendador (vista de tarjeta, clic en CTA) que se retroalimenta al asistente para refinar sus sugerencias.
Contexto de conversación
Información extraída del hilo de chat con el asistente: intenciones detectadas, entidades mencionadas, preferencias expresadas.
Nota de alcance E1: La navegación integrada entre asistente y recomendador requiere que ambos sistemas compartan la infraestructura de sesión (Redis) desde E1. La retroalimentación de señales al motor de recomendación para personalización en tiempo real es parte de etapas posteriores; en E1 las señales se capturan y almacenan para análisis futuro.
