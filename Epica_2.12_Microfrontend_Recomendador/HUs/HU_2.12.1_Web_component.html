Historia de Usuario · HU 2.12.1

Web component / micro-frontend de recomendaciones

Épica 2.12 · Micro-frontend Recomendador · Etapa 1 (Desarrollo)

Enunciado

COMO

equipo web de CCB responsable de integrar el recomendador en el portal.

QUIERO

un componente incrustable (web component / micro-frontend) que pueda integrar en cualquier página del portal CCB.

PARA

integrar el recomendador sin acoplar su código al portal base y facilitar actualizaciones independientes.

1. Criterios de Aceptación E1 – Desarrollo
  1. Componente incrustable e integración en ambiente de desarrollo
    • El componente se publica como Custom Element estándar (<ccb-recomendador>), compatible con cualquier framework (React, Angular, Vue, vanilla JS).
    • Se integra mediante una sola línea de script: <script src="https://cdn.ccb.org.co/recomendador/v1/bundle.js"></script>.
    • Soporta atributos HTML configurables: user-id, canal, limite, tema.
    • Validado en ambiente de desarrollo con integración real en al menos una página del portal.
  2. Contrato de integración (API) definido y estable
    • El contrato de API del componente (atributos de entrada, eventos de salida, métodos públicos) está documentado y versionado.
    • Eventos de salida del componente (via Custom Events):
      • ccb-rec-loaded: componente cargado y recomendaciones disponibles.
      • ccb-rec-cta-click: usuario hizo clic en un CTA de recomendación.
      • ccb-rec-error: error en la carga de recomendaciones.
    • El contrato de API no cambia de forma breaking sin incremento de versión mayor.
  3. Rendimiento y experiencia
    • Tiempo de carga inicial del componente: ≤ 2 segundos (p95) en conexión estándar.
    • Lazy loading: el componente solo solicita recomendaciones cuando entra en el viewport (Intersection Observer).
    • Responsive con breakpoints consistentes con el Design System del portal CCB.
    • Estado de carga con skeleton/spinner; estado de error con mensaje amigable.
  4. Seguridad y autenticación
    • El componente usa OAuth2 como estándar de autorización para acceder al API de recomendaciones; el token se pasa como atributo o mediante evento de handshake con el portal padre.
    • Los outputs del componente están sanitizados anti-XSS (nunca se inserta HTML no controlado del API en el DOM).
    • Estilos aislados mediante Shadow DOM o CSS scoping para evitar colisiones con el portal padre.
    • Cabeceras CSP (Content Security Policy) compatibles con el componente.
  5. Fallback operativo
    • Si el API de recomendaciones no está disponible, el componente muestra un estado de fallback configurable (mensaje, banner alternativo o simplemente ocultarse).
    • El fallback no genera errores en consola visibles al usuario final.
    • El componente emite el evento ccb-rec-error para que el portal padre pueda reaccionar.
  6. OAuth2 y estándar de autorización
    • El flujo de autenticación usa OAuth2 Authorization Code + PKCE para entornos browser.
    • El token de acceso tiene scope mínimo necesario para leer recomendaciones (rec:read).
    • Renovación automática de token (refresh) sin interrumpir la sesión del usuario en el portal.
2. Tecnología Propuesto E1
Web Component (Custom Elements v1)
  • Estándar W3C: Custom Elements v1 + Shadow DOM v1.
  • Bundle independiente versionado (p. ej. Lit, Stencil o vanilla WC).
  • Distribución via CDN propio CCB: cdn.ccb.org.co/recomendador/v{N}/bundle.js.
  • PostMessage API para comunicación bidireccional con el portal padre.
Integración con Motor / API
  • El componente consume el API REST del Motor de Recomendación con autenticación OAuth2.
  • Intersection Observer para lazy loading; no llama al API hasta que el componente es visible.
  • Timeout configurable (default 3s); si supera, activa fallback.
Seguridad
  • Shadow DOM: estilos y DOM del componente aislados del portal padre.
  • Sanitización de contenido del API antes de insertar en DOM (DOMPurify o equivalente).
  • OAuth2 + PKCE para el flujo de autorización browser-side.
  • CSP headers: script-src incluye CDN del componente.
Observabilidad del Componente
  • Métricas de cliente: tiempo de carga, tasa de error, tasa de fallback activado.
  • Custom Events para integración con analítica del portal padre.
  • Logging de errores a endpoint de observabilidad del Motor (sin exponer datos del usuario).
3. Flujo Técnico y de Datos E1 – Desarrollo
ACTOR

Equipo web de CCB que integra el componente en el portal; usuario final que visualiza las recomendaciones.

PARTICIPANTES
  • Usuario / Actor
  • Motor / API (recomendaciones)
  • Frontend / UI (portal padre + componente)
INTEGRACIÓN MÍNIMA
<script src="cdn.ccb.org.co/
  recomendador/v1/bundle.js"
  defer></script>

<ccb-recomendador
  canal="portal"
  limite="5">
</ccb-recomendador>
Pasos del Flujo (según Diagrama de Secuencia)
  1. Componente incrustable e integración: el Usuario/Actor (portal) incluye el script del CDN; el Motor/API confirma disponibilidad del bundle.
  2. Contrato de integración (API): el Frontend/UI (portal padre) recibe el contrato de API del componente (atributos, eventos, métodos); el Motor establece el contrato estable y versionado.
  3. Rendimiento y experiencia: el Motor/API solicita recomendaciones al Frontend/UI cuando el componente entra en viewport (lazy loading); la respuesta se renderiza en ≤ 2s.
  4. Seguridad y autenticación: el Frontend/UI valida el token OAuth2 con el Motor; el Motor verifica scope y devuelve datos autorizados.
  5. Fallback operativo: si el Motor no responde en el timeout configurado, el Frontend/UI activa el fallback y emite ccb-rec-error.
  6. OAuth2 — finalización: el Motor confirma al usuario/actor el estándar de autorización utilizado; el componente queda operativo con token activo.
Ejemplo de Uso del Componente
<!-- Integración en portal CCB -->
<script src="https://cdn.ccb.org.co/recomendador/v1/bundle.js" defer></script>

<ccb-recomendador
  canal="portal-empresas"
  limite="5"
  tema="light">
</ccb-recomendador>

<script>
  const rec = document.querySelector('ccb-recomendador');
  rec.addEventListener('ccb-rec-cta-click', (e) => {
    // El portal padre captura el clic y puede reaccionar
    analytics.track('rec_cta_click', e.detail);
  });
  rec.addEventListener('ccb-rec-error', (e) => {
    // Fallback a nivel de portal
    console.warn('Recomendador no disponible', e.detail);
  });
</script>
4. Diseño — Qué ve el Usuario Componente embebido en portal
Estado de Carga
  • Skeleton loader con el mismo layout que las tarjetas de recomendación.
  • Sin flash de contenido sin estilo (FOUC) — estilos en Shadow DOM.
  • El portal no "salta" mientras carga el componente (reserva de espacio).
Estado Operativo
  • Tarjetas de recomendación con nombre, razones, badges y CTA.
  • Responsive: 1 columna en móvil, 2-3 en desktop.
  • Fallback: mensaje configurable o elemento oculto (display:none) si el API falla.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Carga exitosa en portal React Componente renderiza recomendaciones en ≤ 2s; evento ccb-rec-loaded disparado.
2 Lazy loading (componente fuera de viewport) No se llama al API hasta que el componente entra en viewport; cero requests antes.
3 API no disponible Componente muestra fallback configurado; evento ccb-rec-error disparado; sin errores en consola.
4 Token OAuth2 expirado Refresh automático del token; recomendaciones se cargan sin interrumpir al usuario.
5 Aislamiento de estilos Los estilos del componente no afectan el CSS del portal padre; los estilos del portal no penetran el Shadow DOM.
6 Sanitización XSS Si el API devuelve texto con <script>, el contenido no se ejecuta; se renderiza como texto plano.
7 Responsive en móvil Componente se renderiza en 1 columna en viewport < 576px; sin scroll horizontal.
6. Glosario
Custom Element
Estándar W3C que permite definir nuevas etiquetas HTML con comportamiento propio; base de los Web Components.
Shadow DOM
Árbol DOM encapsulado e independiente del documento principal; los estilos dentro no afectan ni son afectados por el exterior.
Lazy loading
Técnica que difiere la carga de recursos hasta que son necesarios (p. ej., hasta que el componente es visible en el viewport).
OAuth2 + PKCE
Flujo de autorización seguro para aplicaciones de navegador; PKCE previene ataques de intercepción del código de autorización.
PostMessage API
Mecanismo del navegador para comunicación entre documentos de diferente origen (portal padre ↔ componente).
CDN
Red de distribución de contenido; sirve el bundle del componente desde nodos cercanos al usuario para reducir latencia.
Micro-frontend
Arquitectura que divide el frontend en módulos independientes con despliegue autónomo; el web component es la implementación técnica de este concepto.
Nota de alcance E1: En E1 el componente se integra y valida en ambiente de desarrollo con al menos una página del portal CCB. La integración en múltiples páginas y canales (móvil, kiosco) es parte de etapas posteriores. El contrato de API v1 debe mantenerse estable durante todo E1 para no bloquear al equipo de portal.
