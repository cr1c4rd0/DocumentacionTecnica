Historia de Usuario · HU 2.5.2

Solicitud de identificación mínima y progresiva cuando no se pueda identificar

Épica 2.5 · Resolución de Identidad Omnicanal · Etapa 1 (Desarrollo)

Enunciado

COMO

usuario que interactúa con el Recomendador de servicios de la CCB en cualquier canal sin identificación previa o con identificación insuficiente.

QUIERO

que el sistema detecte que no puede identificarme, solicite el dato mínimo necesario de forma no intrusiva y, con ese dato, resuelva mi identidad para poder recomendarme servicios relevantes.

PARA

recibir recomendaciones personalizadas sin tener que proporcionar más información de la estrictamente necesaria, con una experiencia de baja fricción y con dos rutas claras de respuesta.

1. Criterios de Aceptación E1 – Desarrollo
  1. Detección de no-identificación
    • El sistema detecta cuándo no es posible identificar al usuario con los datos disponibles (sin cookie válida, sin sesión, sin identificador en el canal).
    • También aplica cuando la identificación es posible pero el perfil es insuficiente para generar recomendaciones personalizadas de calidad.
  2. Dato mínimo según el caso
    • El sistema determina cuál es el dato mínimo necesario según el canal y contexto:
      • Web: correo electrónico o NIT.
      • WhatsApp/Chat: número de teléfono o correo.
      • Presencial: cédula o NIT capturado por el asesor.
    • Nunca se solicitan más de 2 datos en la primera interacción.
  3. Solicitud no intrusiva del dato
    • El dato se solicita mediante un banner o modal no intrusivo (no bloquea el contenido principal).
    • Mensaje claro sobre por qué se solicita el dato y qué mejora en la experiencia.
    • Opción explícita de "Continuar sin identificarme" siempre visible.
  4. Resolución de identidad con el dato mínimo
    • Con el dato proporcionado, el sistema intenta match contra CRM u otros identificadores (SIREP).
    • Si el match es exitoso: se activa el perfil personalizado y se generan recomendaciones Top-N.
    • Si el match falla: se genera un perfil anónimo mejorado con el dato como señal y se ofrecen recomendaciones de segmento.
  5. Registro y prueba del flujo end-to-end
    • Cada intento de identificación progresiva queda registrado: canal, dato solicitado, resultado (match / no match), latencia.
    • Script/test de integración que valida el flujo completo de no-identificación → solicitud → resolución → recomendación.
  6. Dos rutas claras
    • Ruta A — Identificarse: usuario proporciona dato mínimo → match → recomendaciones personalizadas.
    • Ruta B — Continuar anónimo: usuario declina → recomendaciones genéricas por popularidad/segmento detectado.
    • En ambas rutas el usuario recibe recomendaciones inmediatamente.
2. Tecnología Propuesto E1
Detección y Decisión
  • Lógica de detección de no-identificación en el Motor/API.
  • Regla de selección del dato mínimo según canal (configurable en YAML).
  • Umbral de confianza configurable: si score < threshold → activar flujo progresivo.
Integraciones
  • CRM Dynamics: match del dato mínimo con registros existentes.
  • SIREP: validación cruzada si CRM no tiene coincidencia.
  • Motor/API: activa perfil personalizado o genérico según resultado.
  • Frontend/UI & Asistente/Chat: presenta el banner/modal no intrusivo.
Perfil Anónimo Mejorado
  • Si el match falla pero se tiene el dato: se crea un perfil temporal enriquecido.
  • Sesión anónima con ID temporal que puede vincularse a identidad real en interacciones futuras.
  • Progressive profiling: cada dato adicional mejora la calidad de las recomendaciones.
Privacidad
  • Datos mínimos nunca se persisten sin consentimiento explícito.
  • En E1: datos seudonimizados o mockeados.
  • Cumplimiento Ley 1581 en todos los canales.
3. Flujo Técnico y de Datos E1 – Desarrollo
Pasos del Pipeline de Identificación Progresiva
  1. Inicio: usuario interactúa con el sistema. Se evalúa si hay identificación disponible (HU 2.5.1 primera parte).
  2. Detectar no-identificación: SIREP verifica si los identificadores disponibles son insuficientes para resolver identidad con certeza.
  3. Determinar dato mínimo: Motor/API selecciona qué dato solicitar según el canal (Frontend/UI).
  4. Solicitar dato: Frontend/UI o Asistente/Chat presenta el banner/modal no intrusivo con dos rutas claras.
  5. Resolver identidad: con el dato proporcionado, se hace match contra CRM y otros sistemas; resultado persiste.
  6. Registrar flujo end-to-end: métricas de resolución guardadas en log estructurado.
  7. Banner o modal no intrusivo: confirmación al usuario del resultado y mejora en personalización.
  8. Dos rutas claras: Ruta A (identificado → recomendaciones personalizadas) o Ruta B (anónimo → recomendaciones genéricas).
Ejemplo de Solicitud Mínima (Web)
// Banner no intrusivo — HTML simplificado
{
  "tipo": "banner_identificacion",
  "canal": "web",
  "mensaje": "Para recomendarte servicios a tu medida, ingresa tu correo o NIT.",
  "dato_solicitado": "email_o_nit",
  "rutas": [
    { "id": "A", "label": "Mejorar mis recomendaciones", "accion": "mostrar_campo_input" },
    { "id": "B", "label": "Continuar sin identificarme", "accion": "recomendaciones_genericas" }
  ],
  "bloquea_contenido": false
}
4. Diseño — Banner / Modal No Intrusivo Baja fricción
Principios de UX
  • No interrumpe el flujo del usuario: aparece como banner inferior o lateral.
  • Mensaje en lenguaje positivo: "Mejora tus recomendaciones con un solo dato."
  • Campo de entrada mínimo (un solo input) + botón CTA + botón de cierre visible.
  • Compatible con accesibilidad WCAG 2.1 AA.
Comportamiento Post-Dato
  • Si match exitoso: banner reemplazado por confirmación y recomendaciones se actualizan en tiempo real.
  • Si match fallido: recomendaciones de segmento mejoradas, sin exponer el error al usuario.
  • Si usuario declina: banner desaparece por 24 horas (no vuelve a molestar en la misma sesión).
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Detección de no-identificación Usuario sin cookie ni sesión activa → sistema detecta no-identificación y activa flujo progresivo.
2 Solicitud de dato mínimo (web) Banner aparece solicitando email/NIT; no bloquea el contenido principal; botón "Continuar sin identificarme" visible.
3 Match exitoso con email Email ingresado coincide con CRM; recomendaciones personalizadas activadas en ≤ 300 ms; log registra resultado=match.
4 Match fallido Dato no coincide con CRM ni SIREP; recomendaciones de segmento generadas; log registra resultado=no_match.
5 Ruta B — Continuar anónimo Usuario declina → recomendaciones genéricas mostradas; banner no vuelve a aparecer en la misma sesión.
6 Máximo 2 datos solicitados En ningún caso el sistema solicita más de 2 datos en la primera interacción.
6. Glosario
Identificación Progresiva
Estrategia de recopilación de datos de usuario de forma incremental y no intrusiva, solicitando sólo el mínimo necesario en cada interacción.
Dato Mínimo
El único dato que el sistema solicita al usuario para intentar resolver su identidad (ej. email, NIT, teléfono).
Perfil Anónimo Mejorado
Perfil temporal enriquecido con el dato mínimo cuando el match con CRM/SIREP falla; mejora recomendaciones sin identificar al usuario.
Banner No Intrusivo
Componente de UI que solicita información sin bloquear el contenido principal ni forzar al usuario a responder.
Progressive Profiling
Técnica para enriquecer el perfil de un usuario gradualmente a través de múltiples interacciones, pidiendo un dato diferente cada vez.
Nota de alcance E1: El banner/modal puede ser una implementación simplificada para validación. Lo esencial es el flujo de detección → solicitud mínima → resolución → dos rutas, con registro end-to-end. El diseño visual final se refina en etapas de producción.
