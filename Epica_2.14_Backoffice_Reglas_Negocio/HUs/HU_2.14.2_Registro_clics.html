Registro de clics y navegación

Épica 2.14 · Backoffice y Reglas de Negocio

Enunciado

COMO

CCB (equipo de negocio y analítica del Recomendador de Servicios)

QUIERO

capturar de forma automática, consentida y trazable todos los eventos de interacción del usuario con el recomendador (clics, vistas, descartes, navegación)

PARA

alimentar el ciclo de mejora continua del motor, habilitar analítica de comportamiento y soportar el reentrenamiento de modelos de ML, sin impactar la experiencia del usuario

1. Criterios de Aceptación Tracking
  1. Esquema de eventos mínimo y obligatorio (por sesión)
    • Se capturan obligatoriamente los siguientes tipos de evento por sesión:
      1. recommendation_shown — recomendación mostrada al usuario.
      2. recommendation_clicked — clic sobre una tarjeta de recomendación.
      3. recommendation_dismissed — usuario descarta/oculta una recomendación.
      4. filter_applied — usuario aplica un filtro en el catálogo.
      5. service_detail_viewed — usuario abre el detalle de un servicio.
    • Cada evento incluye: event_type, session_id, user_key_pseudo (o anonymous_id), service_id, timestamp, canal, position_in_list, version_modelo.
  2. Consentimiento y cumplimiento (control de captura)
    • El SDK de tracking verifica el flag de consentimiento antes de registrar eventos de usuarios identificados.
    • Para usuarios anónimos, solo se registra session_id temporal; no se persiste PII.
    • Si el consentimiento se revoca, los eventos del usuario se anonimiza retroactivamente dentro de las 48 h siguientes (proceso batch nocturno).
    • Cumplimiento con Ley 1581 / GDPR: no se registran datos de navegación fuera del contexto del recomendador.
  3. Disponibilidad para analítica y ML (listas para consumo)
    • Los eventos están disponibles para consulta analítica en ≤ 15 min desde su generación (near-real-time).
    • Para pipelines de ML, los eventos se consolidan en tablas Parquet/Delta en el Data Lake, particionadas por fecha y tipo de evento.
    • Se ofrece exportación de eventos en formato CSV o JSON desde el panel de backoffice.
    • Query engine compatible: Athena, BigQuery o Spark.
  4. Calidad de datos: deduplicación y orden
    • El pipeline de ingesta detecta y descarta eventos duplicados usando event_id único (UUID por evento).
    • Los eventos se ordenan por timestamp en el proceso de consolidación; se toleran llegadas tardías de hasta 5 min.
    • Alertas automáticas si el volumen de eventos cae por debajo del umbral esperado (posible pérdida de tracking).
  5. Rendimiento y resiliencia (no impactar UX)
    • La captura de eventos es asíncrona y no bloqueante; latencia de ingesta ≤ 100 ms desde la acción del usuario.
    • Si el servicio de tracking no responde, la UI no se degrada; el evento se encola localmente y se reintenta (fire-and-forget con buffer).
    • El SDK no impacta el tiempo de carga de la página en más de 50 ms (medido en lighthouse).
    • Retention configurable por tipo de evento (defecto 12 meses); archivado automático tras periodo de retención.
2. Tecnología Stack propuesto
SDK de Tracking (Cliente)
  • Biblioteca JavaScript ligera (< 10 KB gzip) con verificación de consentimiento integrada.
  • Buffer local en localStorage para eventos no entregados.
  • Auto-retry silencioso con backoff exponencial (máx. 3 intentos).
  • Modo de debug activable para QA (?tracking_debug=true).
Pipeline de Ingesta
  • Ingesta vía Kafka topic events.recomendador (o equivalente AWS Kinesis).
  • Procesamiento con Spark Structured Streaming o AWS Glue.
  • Deduplicación por event_id en ventana de 5 min.
  • Almacenamiento en Data Lake particionado por date / event_type.
Almacenamiento y Consulta
  • Tablas Parquet/Delta en S3 o ADLS para eventos históricos.
  • Query engine: Athena / BigQuery / Databricks SQL.
  • Tabla OLAP de resumen diario para dashboards de analítica.
  • API de exportación para descarga en CSV/JSON desde el panel.
Cumplimiento y Calidad
  • Job nocturno de anonimización retroactiva para usuarios que revocan consentimiento.
  • Política de retención configurable por tipo de evento (defecto 12 meses).
  • Alertas de volumen: si recommendation_shown cae > 30% vs. semana anterior → alerta de data quality.
  • Integración con CRM / Dynamics para sincronizar flags de consentimiento.
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: interactúa con el recomendador.
  • Motor / API: esquema de eventos mínimo y obligatorio.
  • Frontend / UI: consentimiento y cumplimiento (control de captura).
  • Formulario: disponibilidad para analítica y ML.
  • Asistente / Chat: calidad de datos: deduplicación y orden.
SECUENCIA PRINCIPAL
  1. Motor / API define esquema de eventos mínimo y obligatorio.
  2. Frontend / UI aplica consentimiento y cumplimiento (control de captura).
  3. Formulario habilita disponibilidad para analítica y ML (tablas listas para consumo).
  4. Asistente / Chat garantiza calidad de datos: deduplicación y orden, devuelve a Motor / API.
  5. Motor / API confirma rendimiento y resiliencia (no impactar UX) al Usuario.
GARANTÍAS DE CALIDAD
  • Latencia de ingesta ≤ 100 ms (p95).
  • Deduplicación por event_id (UUID).
  • Disponibilidad analítica ≤ 15 min tras generación.
  • Retención configurable (defecto 12 meses).
  • No impacto en UX: SDK asíncrono y no bloqueante.

4. Diseño — Panel de Analítica de Eventos
Dashboard de Comportamiento
  • Métricas clave: CTR (clics/vistas), tasa de descarte, tiempo en detalle, funnel de conversión.
  • Segmentación por canal, segmento de usuario y versión del modelo.
  • Gráficos de tendencia por semana/mes con comparativo de versiones del motor.
  • Alertas configurables por umbral de CTR o volumen de eventos.
Exportación de Datos
  • Panel con selector de rango de fechas, tipo de evento y canal.
  • Descarga en CSV o JSON con campos seleccionables.
  • Opción de exportar tabla de resumen diario o eventos individuales.
  • Disponible solo para roles autorizados (RBAC); log de acceso a exportaciones.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Captura del esquema mínimo Cada interacción genera evento con todos los campos obligatorios (event_id, event_type, session_id, timestamp, service_id).
2 Consentimiento no otorgado Evento de usuario sin consentimiento no persiste user_key_pseudo; solo se guarda anonymous_id de sesión.
3 Latencia de ingesta ≤ 100 ms SDK envía evento y recibe ACK del servidor en ≤ 100 ms (p95) sin bloquear la UI.
4 Deduplicación Evento con mismo event_id enviado dos veces produce un solo registro en el Data Lake.
5 Disponibilidad analítica ≤ 15 min Evento generado a las 10:00 es consultable en el query engine antes de las 10:15.
6 Resiliencia ante fallo de tracking Si el endpoint de tracking no responde, la UI no se degrada; el evento se encola y reintenta al recuperarse el servicio.
7 Anonimización por revocación Tras revocación de consentimiento, el job nocturno anonimiza eventos del usuario en ≤ 48 h.
6. Glosario
CTR
Click-Through Rate — razón entre clics y vistas de recomendaciones; métrica clave de engagement.
Fire-and-forget
Patrón de envío asíncrono donde el cliente no espera respuesta; minimiza impacto en UX.
Data Lake
Repositorio de almacenamiento de datos en formatos abiertos (Parquet/Delta) para analítica y ML.
Deduplicación
Proceso de eliminación de eventos duplicados usando un identificador único (event_id UUID).
Anonimización retroactiva
Proceso batch que reemplaza PII en eventos históricos por identificadores no reversibles tras revocación de consentimiento.
Retention
Período configurable durante el cual se conservan los eventos antes de su archivado o eliminación.
Nota de alcance: Esta HU cubre la captura, ingesta, almacenamiento y disponibilidad de eventos de interacción del usuario para analítica y ML. La gestión de reglas de negocio del recomendador se cubre en HU 2.14.1. El uso del feedback explícito (útil/no útil) para reentrenamiento se cubre en Épica 2.16.
