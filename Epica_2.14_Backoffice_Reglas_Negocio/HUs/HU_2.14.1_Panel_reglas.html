Historia de Usuario · HU 2.14.1

Panel de reglas de negocio

Épica 2.14 · Backoffice y Reglas de Negocio

Enunciado

COMO

administrador del Recomendador de Servicios (equipo de negocio CCB, sin perfil técnico profundo)

QUIERO

gestionar desde un panel operativo visual las reglas de negocio del recomendador: exclusión dura, inclusión dura, filtrado contextual y priorización

PARA

controlar qué servicios recomienda el motor, en qué contexto y con qué prioridad, sin intervención del equipo técnico y con trazabilidad completa de cambios

1. Criterios de Aceptación Panel Admin
  1. Panel operativo para crear/editar reglas
    • El panel ofrece CRUD completo (crear, leer, editar, desactivar) de reglas desde interfaz web, sin edición directa de código o configuración.
    • Cada regla tiene: nombre, descripción, tipo, condición, efecto, prioridad, estado (activa/inactiva), autor y fecha.
    • La interfaz ofrece preview del impacto estimado de una regla antes de activarla (cuántos servicios/usuarios afecta).
    • Los cambios de parámetros aplican sin redeploy del motor (hot-reload o feature flag).
  2. Reglas aplicadas efectivamente en la UI
    • Las reglas activas se aplican en la próxima inferencia del motor sin necesidad de reinicio del servicio.
    • El Frontend / UI refleja el efecto de las reglas en los resultados de recomendación en tiempo real.
    • Los endpoints del Motor / API exponen el campo reglas_aplicadas[] en el log de cada inferencia.
  3. Precedencia, conflictos y trazabilidad
    • El sistema detecta conflictos entre reglas (p. ej., un servicio incluido y excluido a la vez) y los reporta antes de guardar.
    • Las reglas tienen un campo de prioridad numérica explícito; ante conflicto, prevalece la regla de mayor prioridad.
    • El historial de cambios registra: campo modificado, valor anterior, valor nuevo, usuario, timestamp y razón del cambio.
    • Los cambios críticos (desactivar regla de exclusión dura) generan notificación al equipo técnico.
  4. Reglas de exclusión dura
    • Permiten excluir servicios específicos del catálogo de recomendaciones para todos los usuarios o para segmentos definidos.
    • Ejemplo: excluir servicio S-099 para el segmento "Empresa grande" durante campaña Q3.
    • La exclusión dura tiene precedencia sobre el ranking del modelo IA y sobre las reglas de inclusión dura de menor prioridad.
  5. Reglas de inclusión dura
    • Permiten forzar la presencia de servicios específicos en las recomendaciones (p. ej., servicios de campaña estratégica).
    • El administrador define posición (slot 1, 2…) o simplemente "asegurar presencia en top-K".
    • Una inclusión dura no puede superar a una exclusión dura de igual o mayor prioridad.
  6. Reglas de filtrado contextual
    • Permiten aplicar filtros dinámicos por atributo: segmento, canal, región, fecha, estado del usuario.
    • Ejemplo: mostrar servicio X solo a usuarios del segmento "MiPyme formalizable" en canal "Web".
    • Los filtros contextuales se evalúan después de la exclusión dura y antes del ranking IA.
  7. Ranking del recomendador IA y reglas de priorización
    • El motor IA produce un ranking base; las reglas de priorización pueden subir o bajar servicios en ese ranking sin eliminarlo.
    • El administrador puede definir boosts positivos (multiplicador de score) o penalizaciones para servicios o categorías.
    • El resultado final se registra con reglas_priorización_aplicadas[] para trazabilidad.
2. Tecnología Stack propuesto
Motor de Reglas
  • Motor de reglas en memoria con carga desde base de datos al inicio y hot-reload vía API.
  • Reglas expresadas en DSL interno o YAML con validación de esquema estricto.
  • Evaluación de orden: Exclusión Dura → Filtrado Contextual → Inclusión Dura → Ranking IA → Priorización.
Panel Admin (Frontend)
  • Interfaz web accesible solo a roles autorizados (RBAC).
  • Formulario de creación/edición con validación en tiempo real.
  • Vista de preview de impacto antes de activar una regla.
  • Tabla de historial de cambios con filtros por usuario, fecha y tipo de regla.
API de Reglas (Backend)
  • API RESTful: GET/POST/PUT/DELETE /rules con autenticación y autorización.
  • Validación de conflictos en backend antes de persistir.
  • Endpoint POST /rules/preview para simular impacto sin activar.
  • Webhook / notificación al equipo técnico en cambios críticos.
Auditoría y Observabilidad
  • Tabla de auditoría: rule_id, campo, valor_anterior, valor_nuevo, usuario, razon, timestamp.
  • Log por inferencia: reglas_aplicadas[] con tipo, id y efecto en el score final.
  • Alertas en Slack/email para cambios críticos (desactivar exclusión dura, cambio de prioridad top).
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: administrador del recomendador.
  • Motor / API: evalúa y aplica reglas en inferencias.
  • Panel Admin: interfaz para CRUD de reglas.
  • Frontend / UI: presenta resultados con reglas aplicadas.
SECUENCIA PRINCIPAL
  1. Admin crea/edita reglas en Panel Admin → Motor / API.
  2. Panel Admin propaga reglas aplicadas efectivamente al Frontend / UI.
  3. Frontend / UI reporta precedencia, conflictos y trazabilidad al Motor / API.
  4. Motor / API aplica reglas de exclusión dura al Panel Admin.
  5. Panel Admin aplica reglas de inclusión dura al Frontend / UI.
  6. Frontend / UI devuelve filtrado contextual al Motor / API.
  7. Motor / API procesa ranking del recomendador IA al Panel Admin.
  8. Panel Admin aplica reglas de priorización y devuelve resultado al Usuario.
ORDEN DE EVALUACIÓN DE REGLAS
  1. Exclusión dura — elimina servicios prohibidos.
  2. Filtrado contextual — aplica condiciones de segmento/canal/fecha.
  3. Inclusión dura — garantiza presencia de servicios obligatorios.
  4. Ranking IA — ordena por score del modelo.
  5. Priorización — aplica boosts/penalizaciones finales.
Ejemplo de Estructura de Regla
{
  "rule_id": "R-045",
  "nombre": "Excluir S-099 para segmento Empresa Grande en Q3",
  "tipo": "exclusion_dura",
  "condicion": {
    "service_ids": ["S-099"],
    "segmentos": ["empresa_grande"],
    "fecha_fin": "2025-09-30"
  },
  "efecto": "exclude",
  "prioridad": 10,
  "estado": "activa",
  "autor": "admin_jperez",
  "timestamp_creacion": "2025-07-01T09:00:00Z",
  "razon": "Capacidad limitada en Q3 para este segmento"
}
4. Diseño — Panel Operativo de Reglas
Vistas Principales del Panel
  • Listado de reglas: tabla con columnas tipo, nombre, estado, prioridad, autor, última modificación. Filtros por tipo y estado.
  • Formulario de regla: campos validados en tiempo real con selector de tipo y condición dinámica.
  • Preview de impacto: muestra número de servicios afectados y usuarios impactados antes de activar.
  • Historial de auditoría: timeline de cambios filtrable por usuario y período.
Indicadores de Conflicto
  • Badge Conflicto en reglas que se contradicen mutuamente.
  • Modal de advertencia antes de guardar: "La regla R-045 entra en conflicto con R-032. ¿Confirmar prioridad?"
  • Indicador visual de cadena de precedencia en la vista de detalle de cada regla.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Crear regla de exclusión dura Servicio S-099 no aparece en top-K para segmento "empresa_grande" tras activar la regla, sin redeploy.
2 Conflicto detectado Al guardar regla que incluye y excluye S-099 simultáneamente, el sistema alerta del conflicto y solicita confirmación de prioridad.
3 Preview de impacto POST /rules/preview devuelve número de servicios y usuarios afectados sin crear la regla.
4 Hot-reload sin redeploy Regla activada a las 10:00 aplica en inferencia de las 10:01 sin reiniciar el servicio.
5 Auditoría de cambio Historial registra: regla editada, campo, valor anterior/nuevo, usuario y timestamp.
6 Orden de precedencia Exclusión dura supera a inclusión dura de menor prioridad; servicio excluido no aparece aunque esté en lista de inclusión.
7 Trazabilidad en inferencia Log de inferencia incluye reglas_aplicadas: ["R-045 exclusion_dura", "R-012 boost_campana"].
8 Notificación por cambio crítico Desactivar una regla de exclusión dura genera notificación automática al equipo técnico.
6. Glosario
Exclusión dura
Regla que elimina un servicio del conjunto de candidatos de forma incondicional, con la más alta precedencia.
Inclusión dura
Regla que garantiza la presencia de un servicio en el top-K independientemente del score del modelo.
Filtrado contextual
Regla que aplica condiciones dinámicas (segmento, canal, fecha) para filtrar candidatos antes del ranking IA.
Hot-reload
Mecanismo que recarga la configuración de reglas en memoria sin reiniciar el servicio.
Boost / Penalización
Multiplicador aplicado al score de un servicio para subir o bajar su posición en el ranking final.
RBAC
Role-Based Access Control — control de acceso basado en roles que restringe el panel admin solo a usuarios autorizados.
Nota de alcance: Esta HU cubre el panel de gestión de reglas de negocio del backoffice del recomendador. El registro de eventos de interacción de usuarios (clics, navegación) se gestiona en HU 2.14.2. El orden de evaluación de reglas es: Exclusión Dura → Filtrado Contextual → Inclusión Dura → Ranking IA → Priorización.
