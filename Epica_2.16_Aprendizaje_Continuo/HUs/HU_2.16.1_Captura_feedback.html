Captura de feedback en recomendador

Épica 2.16 · Aprendizaje Continuo

Enunciado

COMO

usuario del Recomendador de Servicios que recibe recomendaciones personalizadas

QUIERO

indicar con un gesto simple y completamente opcional si una recomendación me resultó útil o no útil

PARA

ayudar al sistema a aprender de mis preferencias reales y hacer que las futuras recomendaciones sean más relevantes para mí, respetando mi privacidad

1. Criterios de Aceptación Feedback Explícito
  1. Feedback explícito por recomendación (útil / no útil)
    • Cada tarjeta de recomendación muestra controles discretos de feedback: 👍 Útil / 👎 No útil (iconografía o texto, visibles pero sin ocupar espacio dominante).
    • El feedback es completamente opcional; no se fuerza interacción ni se penaliza la omisión.
    • El usuario puede cambiar su respuesta (de útil a no útil o viceversa) mientras esté en la misma sesión; el cambio reemplaza el registro anterior.
    • Se acepta un campo de texto libre adicional (máx. 280 caracteres) para comentario opcional sobre el motivo.
  2. Persistencia por usuario cuando exista identidad
    • Si el usuario tiene sesión activa (identificado), el feedback se persiste asociado a user_key_pseudo + recommendation_id.
    • Si el usuario es anónimo, el feedback se asocia a session_id temporal y se descarta al cerrar sesión (sin persistencia de PII).
    • El registro de feedback contiene: feedback_id, recommendation_id, service_id, user_key_pseudo, valor (useful|not_useful), comentario (opcional), timestamp, version_modelo.
  3. Prevención de duplicados y actualización del feedback
    • La combinación (user_key_pseudo, recommendation_id) es única en la tabla de feedback; un segundo envío actualiza el registro existente (UPSERT).
    • El endpoint devuelve el estado actual del feedback (created o updated) para que la UI muestre el estado correcto.
    • Si se actualiza el feedback, el timestamp de modificación se registra por separado del timestamp de creación.
  4. Rendimiento y resiliencia
    • El envío de feedback es asíncrono (fire-and-forget); la UI muestra confirmación visual inmediata sin esperar respuesta del servidor.
    • Latencia de registro: el servidor persiste el feedback en ≤ 500 ms (p95).
    • Si el servidor no responde, el cliente reintenta silenciosamente hasta 3 veces con backoff; si falla, el feedback se descarta sin notificar al usuario (experiencia no interrumpida).
  5. Disponibilidad para analítica y aprendizaje
    • Los registros de feedback están disponibles en el Data Lake en ≤ 15 min para consumo analítico.
    • Agregación diaria automática: suma de útil/no útil por service_id y version_modelo para alimentar el pipeline de reentrenamiento.
    • Los datos de feedback se conservan durante el período de retención configurado (defecto 24 meses) antes de archivado.
    • Si el usuario revoca consentimiento, sus registros de feedback se anonimizan en el siguiente ciclo batch.
2. Tecnología Stack propuesto
API de Feedback
  • POST /feedback con payload mínimo: recommendation_id, service_id, valor, comentario (opcional).
  • Semántica UPSERT en base de datos: crea o actualiza según existencia de (user_key_pseudo, recommendation_id).
  • Respuesta inmediata: HTTP 200/201 con {"status": "created"|"updated"}.
Almacenamiento y Agregación
  • Tabla de feedback en base de datos relacional (PostgreSQL / Cloud SQL) para baja latencia.
  • Replicación a Data Lake (Parquet/Delta) para ML pipeline.
  • Job de agregación diaria: tabla feedback_daily_agg con conteos por servicio y modelo.
  • Retención configurable; job nocturno de anonimización por revocación de consentimiento.
Frontend / UI
  • Componente de feedback discreto en cada tarjeta: botones 👍/👎 + campo de texto colapsable.
  • Confirmación visual inmediata (toast o cambio de estado del botón) sin esperar respuesta del servidor.
  • Estado sincronizado: al recargar la página, la UI refleja el feedback previamente dado (para usuarios identificados).
Privacidad y Cumplimiento
  • No se persiste PII en registros de feedback; se usa user_key_pseudo.
  • Consentimiento verificado en el backend antes de persistir feedback de usuarios identificados.
  • Usuarios anónimos: feedback de sesión descartado al cerrar; no se persiste.
  • Alineado con Ley 1581 y lineamientos internos CCB.
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: usuario del recomendador.
  • Motor / API: captura feedback explícito y lo persiste.
  • Frontend / UI: persistencia por usuario cuando exista identidad.
  • Asistente / Chat: prevención de duplicados y actualización.
HAPPY PATH
  1. Motor / API recibe feedback explícito (útil/no útil) desde el usuario.
  2. Frontend / UI verifica persistencia por usuario cuando exista identidad y consentimiento.
  3. Asistente / Chat aplica prevención de duplicados y actualización del feedback (UPSERT).
  4. Motor / API aplica rendimiento y resiliencia (fire-and-forget).
  5. Frontend / UI confirma disponibilidad para analítica y aprendizaje al Usuario.
BRANCH: SIN IDENTIDAD
  • Si no existe identidad (usuario anónimo) → Error / Fallback: notificar al usuario que el feedback no puede persistirse.
  • Feedback de sesión se mantiene temporalmente en session_id.
  • Al cerrar sesión o expirar token, el feedback temporal se descarta.
Ejemplo de Payload de Feedback
{
  "feedback_id": "fb_3a8c2d",
  "recommendation_id": "rec_u9f3a_S134_20250820",
  "service_id": "S-134",
  "user_key_pseudo": "u_9f3a",
  "valor": "not_useful",
  "comentario": "Ya conozco este servicio, necesito algo nuevo",
  "timestamp_creacion": "2025-08-20T14:35:00Z",
  "timestamp_modificacion": null,
  "version_modelo": "rec-0.3.1",
  "status": "created"
}
4. Diseño — Controles de Feedback en Tarjeta
Componente de Feedback (por tarjeta)
  • Botones discretos en esquina inferior de cada tarjeta: 👍 / 👎.
  • Al hacer clic: confirmación visual inmediata (botón seleccionado se activa, cambia color).
  • Tooltip al pasar el cursor: "¿Esta recomendación es útil para ti?".
  • Campo de texto colapsable (opcional): aparece al dar clic en cualquier botón, permite agregar comentario.
  • Botón "Omitir" para cerrar el campo de texto sin enviar comentario.
Estados del Componente
  • Sin interacción: botones visibles pero tenues (no distrae del contenido principal).
  • Útil seleccionado: 👍 activo (verde), 👎 tenue. Toast: "¡Gracias por tu feedback!"
  • No útil seleccionado: 👎 activo (naranja), 👍 tenue. Toast: "Gracias, mejoraremos tus recomendaciones."
  • Usuario anónimo: botones visibles, al hacer clic se sugiere crear cuenta para guardar preferencias.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Feedback útil con usuario identificado Se crea registro con valor=useful, user_key_pseudo y recommendation_id. La UI muestra 👍 activo con confirmación.
2 Cambio de feedback (UPSERT) Segundo envío con valor=not_useful para mismo recommendation_id actualiza el registro; status=updated, timestamp_modificacion registrado.
3 Usuario anónimo Feedback se asocia solo a session_id; no persiste user_key_pseudo. Al cerrar sesión, el registro temporal se descarta.
4 Latencia de persistencia ≤ 500 ms Servidor confirma persistencia del feedback en ≤ 500 ms (p95).
5 Resiliencia ante fallo del servidor Si el servidor no responde, la UI muestra confirmación inmediata de todas formas; el cliente reintenta silenciosamente hasta 3 veces.
6 Disponibilidad analítica ≤ 15 min Feedback persistido a las 10:00 es consultable en el Data Lake antes de las 10:15.
7 Anonimización por revocación Tras revocación de consentimiento, el job batch anonimiza registros de feedback del usuario en ≤ 24 h.
6. Glosario
Feedback explícito
Señal de preferencia directa del usuario (útil/no útil), en contraste con el feedback implícito (clics, tiempo de visualización).
UPSERT
Operación de base de datos que inserta un registro si no existe o lo actualiza si ya existe, usando una clave única.
Recommendation ID
Identificador único de una recomendación específica entregada a un usuario en un momento dado (user + service + timestamp).
Fire-and-forget
Patrón asíncrono donde el cliente no espera la respuesta del servidor; minimiza el impacto percibido en la UX.
Agregación diaria
Job batch que consolida conteos de feedback por servicio y versión de modelo, generando una tabla de resumen para el pipeline de ML.
Aprendizaje continuo
Ciclo iterativo en el que los datos de feedback capturados alimentan el reentrenamiento del modelo para mejorar su precisión con el tiempo.
Nota de alcance: Esta HU cubre la captura, persistencia y disponibilidad del feedback explícito del usuario (útil/no útil) sobre las recomendaciones recibidas. El uso de ese feedback para reentrenar el modelo y desplegar una nueva versión se gestiona en HU 2.16.2. El feedback es siempre opcional; no se fuerza al usuario a interactuar.
