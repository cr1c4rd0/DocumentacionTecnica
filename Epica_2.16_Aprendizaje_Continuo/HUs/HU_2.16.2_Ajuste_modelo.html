Historia de Usuario · HU 2.16.2

Ajuste de modelo por feedback

Épica 2.16 · Aprendizaje Continuo

Enunciado

COMO

Científico de Datos del Recomendador de Servicios (CCB)

QUIERO

incorporar el feedback explícito e implícito de los usuarios en un ciclo de entrenamiento periódico del modelo, con evaluación rigurosa antes del despliegue

PARA

que el motor de recomendación evolucione con el comportamiento real de los usuarios, mejore sus métricas de negocio y pueda desplegarse o revertirse de forma controlada

1. Criterios de Aceptación Reentrenamiento
  1. Incorporación de feedback en un ciclo de entrenamiento
    • El pipeline de reentrenamiento consume de forma automática:
      • Feedback explícito (útil/no útil) de las últimas N semanas (N configurable, defecto 4).
      • Señales implícitas: clics, vistas, descartes, tiempo de visualización (de HU 2.14.2).
      • Datos históricos de servicios contratados (SIREP).
    • El ciclo de reentrenamiento se ejecuta de forma periódica y automatizada (cron semanal o bajo demanda desde panel admin).
    • Cada ciclo produce un artefacto de modelo versionado con: version_id, fecha_entrenamiento, datos_usados, métricas_offline.
  2. Evaluación antes del despliegue — CCB define rol responsable
    • Antes de desplegar una nueva versión, se ejecuta automáticamente una batería de evaluación offline:
      • Métricas de calidad: precision@K, recall@K, NDCG, coverage, diversity, novelty.
      • Validación de fairness y sesgo: distribución por segmento; alerta si hay diferencias significativas entre grupos.
      • Comparación con versión actual: mejora mínima configurable (defecto: +2% en precision@5).
    • La CCB define un rol responsable de aprobación (Científico de Datos + Product Owner) que revisa el reporte de evaluación antes de autorizar el despliegue.
    • Si las métricas no superan el umbral mínimo, el despliegue se bloquea y se notifica al equipo con el reporte de comparación.
  3. Despliegue controlado y rollback
    • El despliegue de la nueva versión se realiza de forma gradual (canary: 5% → 20% → 100%) con monitoreo de métricas en cada etapa.
    • Si en cualquier etapa del canary las métricas online caen por debajo del umbral configurado, se activa rollback automático a la versión anterior.
    • El rollback manual también está disponible desde el panel admin con un clic.
    • El estado de cada versión del modelo queda registrado: en_prueba, activa, retirada, rollback.
  4. Privacidad/consentimiento y minimización
    • Solo se usan en el entrenamiento datos de usuarios con consentimiento explícito de uso de sus datos para mejora del servicio.
    • Los datos de entrenamiento se trabajan con user_key_pseudo (seudonimizados); no se expone PII en el pipeline de ML.
    • Principio de minimización: solo se incluyen las features estrictamente necesarias para el modelo; se documentan en el artefacto de versión.
    • Los datasets de entrenamiento se eliminan o archivan tras el ciclo, según política de retención de datos de ML.
  5. Rendimiento del motor en producción
    • La nueva versión del modelo no debe degradar la latencia de inferencia: p95 ≤ tope definido en configuración (referencia: 300 ms).
    • Se monitorean métricas online en producción tras cada despliegue: CTR, tasa de descarte, conversión, latencia p95.
    • Alertas configuradas en el sistema de observabilidad (Grafana/Datadog) para degradación de métricas post-despliegue.
    • Notificación automática al equipo cuando el modelo se actualiza exitosamente en producción.
2. Tecnología Stack propuesto
Pipeline de Reentrenamiento
  • Orquestación con Apache Airflow o equivalente (DAG semanal + on-demand).
  • Tracking de experimentos con MLflow: versiones, parámetros, métricas, artefactos.
  • Feature store actualizado con señales de feedback e interacciones recientes.
  • Framework de entrenamiento: scikit-learn / PyTorch / LightFM según modelo base.
Evaluación y Validación
  • Métricas offline: precision@K, recall@K, NDCG, coverage, diversity, novelty (script reproducible).
  • Validación de fairness: Fairlearn o AIF360 — distribución equitativa entre segmentos.
  • Comparación automática con versión en producción; bloqueo si no supera umbral mínimo.
  • Reporte de evaluación generado automáticamente y enviado al rol responsable para aprobación.
Despliegue y Rollback
  • Estrategia canary deployment: 5% → 20% → 100% con pausa entre etapas.
  • Rollback automático si métricas online caen por debajo de umbral configurado.
  • Rollback manual desde panel admin con registro de motivo.
  • Registro de estado de cada versión: en_prueba | activa | retirada | rollback.
Observabilidad en Producción
  • Dashboards en Grafana / Datadog: CTR, latencia p95, tasa de descarte, cobertura por segmento.
  • Alertas automáticas por degradación de métricas post-despliegue.
  • Notificación al equipo (Slack/email) en cada actualización exitosa del modelo.
  • Panel de historial de versiones con métricas comparativas offline y online.
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: Científico de Datos.
  • Motor / API: incorporación de feedback y rendimiento en producción.
  • Panel Admin: evaluación antes del despliegue y rol responsable.
  • Frontend / UI: despliegue controlado y rollback.
  • Formulario: privacidad/consentimiento y minimización.
SECUENCIA PRINCIPAL
  1. Motor / API lanza ciclo de incorporación de feedback en entrenamiento.
  2. Panel Admin ejecuta evaluación; CCB define rol responsable para aprobación del despliegue.
  3. Frontend / UI gestiona despliegue controlado (canary) y rollback si es necesario.
  4. Formulario verifica privacidad/consentimiento y minimización de datos de entrenamiento.
  5. Motor / API monitorea rendimiento del motor en producción y notifica al equipo.
CICLO DE APRENDIZAJE
  1. Recolección: feedback + interacciones + SIREP.
  2. Entrenamiento: pipeline automatizado (Airflow + MLflow).
  3. Evaluación: métricas offline + fairness + umbral mínimo.
  4. Aprobación: rol responsable revisa reporte.
  5. Despliegue: canary gradual con monitoreo online.
  6. Monitoreo: alertas y métricas en producción.
  7. Rollback: automático o manual si hay degradación.

4. Diseño — Panel de Gestión de Versiones del Modelo
Vista de Versiones del Modelo
  • Tabla con columnas: versión, fecha, estado, precision@5, delta vs. anterior, aprobado por.
  • Badge de estado: Activa En prueba Retirada Rollback.
  • Botón "Aprobar despliegue" disponible para versiones en_prueba que superaron umbral.
  • Botón "Rollback manual" disponible para la versión activa.
Reporte de Evaluación (por versión)
  • Métricas offline con comparativo vs. versión anterior (tabla + gráfico de barras).
  • Sección de fairness: distribución por segmento, alertas de sesgo.
  • Lista de features usadas en el entrenamiento.
  • Enlace a experimento en MLflow para trazabilidad completa.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Ciclo de entrenamiento automático El DAG de Airflow ejecuta el pipeline semanal y produce artefacto de modelo versionado con métricas offline registradas en MLflow.
2 Bloqueo por umbral no superado Si precision@5 no mejora ≥ 2% vs. versión actual, el despliegue queda bloqueado y se notifica al equipo con reporte comparativo.
3 Aprobación y despliegue canary Rol responsable aprueba desde panel; la versión se despliega al 5% del tráfico. Tras métricas estables, escala a 20% y luego a 100%.
4 Rollback automático Si CTR cae > umbral configurado durante canary, el sistema revierte automáticamente a la versión anterior y notifica al equipo.
5 Fairness sin sesgo El reporte de evaluación no muestra alertas de sesgo entre segmentos; la diferencia de precision entre grupos es ≤ umbral configurado.
6 Privacidad en entrenamiento El dataset de entrenamiento no contiene PII directa; solo user_key_pseudo. Solo incluye datos de usuarios con consentimiento activo.
7 Latencia en producción no degradada La nueva versión del modelo mantiene latencia de inferencia p95 ≤ tope configurado tras despliegue completo.
6. Glosario
NDCG
Normalized Discounted Cumulative Gain — métrica de ranking que penaliza elementos relevantes en posiciones bajas de la lista.
Canary Deployment
Estrategia de despliegue gradual que expone la nueva versión a un porcentaje pequeño del tráfico antes de escalar al 100%.
Fairness / Fairlearn
Conjunto de técnicas y herramientas para detectar y mitigar sesgo en modelos de ML entre diferentes grupos demográficos o de negocio.
Rollback
Reversión del sistema a una versión anterior del modelo, ya sea automática (por umbral) o manual (desde panel admin).
Feature Store
Repositorio centralizado de variables (features) listas para entrenamiento e inferencia, actualizado con señales de feedback e interacciones.
MLflow
Plataforma de tracking de experimentos de ML que registra parámetros, métricas y artefactos por cada ciclo de entrenamiento.
Seudonimización
Sustitución de PII por una clave no reversible (user_key_pseudo) que permite el análisis sin exponer datos personales.
Nota de alcance: Esta HU cubre el ciclo completo de reentrenamiento del modelo con feedback, evaluación rigurosa, despliegue controlado (canary) y rollback. La captura del feedback explícito del usuario se gestiona en HU 2.16.1. El registro de eventos de interacción para ML se gestiona en HU 2.14.2. La CCB define el rol responsable de aprobación del despliegue de cada nueva versión.
