¿Y si no existo? — Recomendación para usuario sin historial

Épica 2.1 · Motor de Recomendación Personalizada · Cold Start

Enunciado

COMO

usuario cuya identificación no existe en el CRM o en el historial de clientes que han navegado anteriormente en ccb.org.co.

QUIERO

que el sistema me ofrezca recomendaciones proactivas basadas en información mínima disponible (sector, intereses declarados, navegación o atributos generales), o aleatorias que ayuden a ajustar el perfil progresivamente.

PARA

recibir ayuda inmediata aunque no tenga un perfil en la CCB, y que el sistema pueda aprender mi tipo de cliente, preferencias e intereses desde el inicio.

Objetivo funcional: Habilitar un flujo alterno de recomendación cuando el CRM no devuelve coincidencias, evitando bloqueos y permitiendo ofrecer valor incluso a usuarios nuevos, externos o sin registro previo.
1. Criterios de Aceptación Definition of Done
  1. CA 1 — Detección de no existencia
    • El sistema valida el ID del usuario contra la réplica del CRM.
    • Si no existe, activa el "modo recomendación mínima viable" (no personalizada).
  2. CA 2 — Ofrecer alternativas claras
    • Cuando no se encuentra el usuario, el asistente ofrece:
      1. Continuar sin identificación (modo invitado).
      2. Proveer algunos datos (p. ej.: sector, tipo de empresa, interés).
      3. Acceder al flujo de registro (solo si negocio lo aprueba).
  3. CA 3 — Recomendación proactiva basada en datos no sensibles
    • El motor genera recomendaciones usando:
      • Atributos declarados por el usuario (self-reported).
      • Populares del portafolio.
      • Servicios gratuitos.
      • Servicios más demandados por segmento general.
      • Servicios introductorios para emprendedores o nuevos empresarios.
  4. CA 4 — Cumplimiento normativo
    • No se infiere identidad.
    • No se construye perfil sin consentimiento.
    • Cumplimiento de la Ley 1581 de protección de datos.
    • Decisiones trazadas y auditables.
  5. CA 5 — Lógica explícita
    • El recomendador explica claramente: "Te recomiendo estos servicios porque aún no encuentro tu información en nuestra base de datos, pero estos son los más útiles para empresarios como tú."
  6. CA 6 — Métricas obligatorias
    • % de usuarios recomendados en modo invitado.
    • % de usuarios que pasan a registro.
    • CTR en modo invitado.
    • Efectividad vs. recomendaciones personalizadas.
  7. CA 7 — Integración con el motor principal
    • El sistema cambia automáticamente de "modo invitado" a "modo personalizado" cuando el usuario ingresa su ID válido.
    • Persistencia mínima del estado de transición.
2. Tecnología Propuesto E1
Detección y Fallback
  • Validación de ID contra réplica CRM en tiempo real.
  • Activación automática de modo invitado con degradación segura.
  • Flag is_cold_start en cada request de recomendación.
Estrategias Cold Start
  • Popularidad por segmento general: top servicios del portafolio.
  • Self-reported: datos declarados por el usuario (sector, tipo empresa).
  • Servicios introductorios: enfocados a nuevos empresarios.
  • Servicios gratuitos: priorizados sin riesgo de entrada.
Transición a Modo Personalizado
  • Umbral configurable de interacciones / señales acumuladas.
  • Cambio automático cuando el usuario provee ID válido de CRM.
  • A/B testing entre estrategias cold start para optimización.
Privacidad y Trazabilidad
  • Sin PII en el payload de recomendaciones para usuarios sin registro.
  • Decisiones auditables: razón de cada recomendación registrada.
  • Cumplimiento Ley 1581 — consentimiento previo para perfilado.
3. Flujo Técnico y de Datos
Pasos del Flujo Cold Start
  1. Entrada: usuario accede al portal (anónimo o con ID no reconocido).
  2. Validación CRM: el backend consulta la réplica del CRM con el ID proporcionado.
  3. Detección cold start: si no existe coincidencia, se activa mode=cold_start.
  4. Oferta de alternativas: el sistema presenta opciones al usuario (modo invitado, ingresar datos, registrarse).
  5. Generación de candidatos: usando datos self-reported + populares por segmento general + servicios gratuitos + servicios introductorios.
  6. Re-ranking y reglas: aplicar filtros de elegibilidad básicos y boost de gratuitos.
  7. Respuesta: top-N recomendaciones con razón explícita del modo invitado.
  8. Captura de señales: las interacciones del usuario van construyendo su perfil para la transición al modo personalizado.
4. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 ID no encontrado en CRM Se activa modo invitado; se generan recomendaciones basadas en popularidad general.
2 Usuario provee sector y tipo de empresa Las recomendaciones se ajustan al segmento self-reported; score mejora vs. baseline random.
3 Explicación clara en modo invitado Cada recomendación incluye razón legible indicando el motivo del modo sin historial.
4 Transición a modo personalizado Al ingresar ID válido del CRM, el sistema cambia automáticamente a recomendaciones personalizadas.
5 Sin PII en logs El payload de decisiones no contiene datos personales identificables del usuario invitado.
6 Métricas de cohorte cold start El dashboard reporta CTR, % registro y efectividad vs. personalizadas para el segmento invitado.
5. Glosario
Cold Start
Problema de recomendación cuando no existe historial previo del usuario o el ítem.
Modo Invitado
Estado del sistema cuando el usuario navega sin identificación reconocida; usa popularidad y datos declarados.
Self-reported
Atributos que el usuario declara voluntariamente (sector, tipo empresa, interés) sin que el sistema los infiera.
Modo Personalizado
Estado del sistema cuando el usuario tiene perfil en CRM y suficientes señales para recomendación individual.
anon_id
Identificador de sesión temporal sin PII, para rastrear señales del usuario invitado.
