Entrenar motor con 6M+ clientes / 24M+ interacciones

Épica 2.1 · Motor de Recomendación Personalizada · Etapa 1 (Desarrollo)

Enunciado

COMO

científico de datos de la CCB.

QUIERO

entrenar y versionar un motor de recomendación usando el histórico de ≥6 millones de clientes y ≥24 millones de interacciones (consultas, consumos, clics, conversiones).

PARA

optimizar CTR/CR y dejar una base reproducible para experimentación y despliegue futuro.

1. Criterios de Aceptación E1 – Desarrollo
  1. Dataset listo para entrenamiento a escala
    • Existe un pipeline reproducible (script/notebook + orquestación simple) que construye datasets de entrenamiento/validación/test desde fuentes dev/seudonimizadas.
    • El pipeline maneja desbalance (p. ej., negative sampling y/o class weights) y fugas de información (split temporal por usuario).
  2. Feature Store mínima operativa
    • Catálogo/tabla(s) con:
      • Features de usuario (segmento, rol, antigüedad, señales de interés).
      • Features de servicio (categoría, elegibilidad, campaña, popularidad).
      • Features de interacción (frecuencia/recencia, canales).
    • Esquema versionado (feature_group, version, created_at) y diccionario de datos.
  3. Entrenamiento y versionado del modelo
    • Entrena al menos un enfoque baseline (p. ej., factorización matricial/LightFM o Two-Tower con embeddings) y uno híbrido (contenido + reglas).
    • Artefactos versionados: modelo.pkl (o formato equivalente), requirements.txt, config.yml, commit/tag.
    • Registra métricas offline: AUC/ROC, MAP@K, NDCG@K, Recall@K, sin exigir umbrales en E1 (solo medición y reporte).
  4. Evaluación offline y "shadow" planificada
    • Reporte comparativo baseline vs. híbrido con tablas/gráficos e intervalos por bootstrap o k-fold temporal.
    • Se documenta un plan de evaluación online en "shadow mode" (sin afectar usuarios) para fase siguiente; en E1 no se ejecuta A/B productivo.
  5. Escalabilidad demostrada en dev
    • Evidencia de tiempos de entrenamiento y uso de memoria para un subconjunto representativo (p. ej., 10–20% del total) y estimación lineal para el total.
    • Batch inference genera top-K por usuario para un lote ≥100k usuarios con latencia por usuario reportada (p95/p99).
  6. Trazabilidad, ética y privacidad
    • Seudonimización de identificadores; no se persiste PII en artefactos.
    • Bitácora: data_versions, feature_versions, model_version, seed, hyperparams, latencias.
    • Documento de riesgos de sesgo + mitigaciones iniciales (diversidad mínima y exclusión de atributos sensibles).
2. Tecnología Propuesto E1
Procesamiento
  • Python + PySpark/Polars (según volumen disponible en dev).
  • Almacenamiento Parquet/Delta.
Modelado
  • Baseline: LightFM/ALS (implícito) o Two-Tower (TensorFlow/PyTorch).
  • Híbrido: similitud de contenido (embeddings/texto) + reglas de negocio (elegibilidad, no-repetición).
Feature Store & Versionado
  • Tablas particionadas por fecha/segmento; catálogo (Glue/HCatalog/Delta catalog) en dev.
  • Git + etiquetas; opcional MLflow para run_id, métricas y artefactos.
Observabilidad
  • Logs estructurados (JSON).
  • Métricas guardadas en CSV/Parquet y gráfico HTML/PNG.
3. Flujo Técnico y de Datos E1 – Desarrollo
INICIO

Job manual/orquestado (cron dev) que ejecuta: feature build → training → evaluation → packaging.

FUENTES (dev/seudonimizadas)
  • CRM Dynamics: segmento, rol, antigüedad, estado cliente.
  • SIREP: interacciones/transacciones (consumos, clics, fechas, canal).
  • Catálogo: metadata, elegibilidad, campañas, categorías.
  • Señales: popularidad, novedad, temporadas.
SALIDA TÉCNICA

Artefactos versionados + reporte con métricas offline, tiempos, consumo, ejemplos de top-K explicados y lista de errores conocidos/riesgos.

Pasos del Pipeline
  1. Ingesta & limpieza: normaliza IDs, fechas, canales; seudonimiza usuario.
  2. Construcción de features: ventanas (7/30/90 días), recency-frequency, embeddings de descripciones.
  3. Split temporal y balanceo: holdout por mes/semana; negative sampling por usuario.
  4. Entrenamiento: baseline + híbrido; grid/random search de hiperparámetros.
  5. Evaluación offline: MAP@10, NDCG@10, Recall@10, AUC; tablas y gráficos.
  6. Empaque: modelo.pkl + config.yml + preprocess.pkl; guardado con hash/versión.
  7. Batch inference (dev): top-K para lote; exclusiones (ya tomados/declinados).
  8. Reporte: notebook/markdown con resultados, costos estimados y próximos pasos.
4. Diseño — Qué ve el Usuario E1 sin front público
Validación E1 (notebook / CSV)
  • Tabla comparativa de modelos y métricas (baseline vs. híbrido).
  • Muestras de recomendaciones por usuario con campos: service_id, nombre, score, razones[], reglas_aplicadas.
No hay UI final en E1
  • La validación es vía notebook de resultados y salida JSON/CSV.
  • El A/B real y la UI de negocio se programan para la siguiente fase.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Split temporal correcto Ninguna interacción del futuro "contamina" el conjunto de entrenamiento.
2 Balanceo Negative sampling activo y configurable; impacto reflejado en métricas.
3 Exclusiones de negocio Post-ranking elimina servicios taken/declined.
4 Reproducibilidad Ejecutar con la misma seed genera métricas dentro de tolerancia.
5 Escalabilidad Lote ≥100k usuarios genera top-K y registra p95/p99 de inferencia.
6. Glosario
CTR / CR
Click-Through Rate / Conversion Rate.
MAP@K / NDCG@K / Recall@K
Métricas de ranking evaluadas en el top-K de recomendaciones.
Negative Sampling
Generación de no-interacciones para balancear clases en entrenamiento.
Two-Tower
Arquitectura de dos torres (usuario/ítem) para aprendizaje de similitud.
Seudonimización
Reemplazo de PII por identificadores no reversibles usados en dev.
Shadow Mode
Ejecución paralela del modelo nuevo sin impactar al usuario final.
Feature Store
Repositorio/tablas con variables de usuario/servicio listas para inferencia, con versión y sello de tiempo.
Nota de alcance E1: Se mide y documenta todo (sin umbrales obligatorios), se deja artefacto versionado, Feature Store mínima y reporte reproducible. El A/B real y la UI de negocio se programan para la siguiente fase.
