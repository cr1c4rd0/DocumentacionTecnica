Historia de Usuario · HU 2.11.1

Redirección al formulario correcto

Épica 2.11 · Integración con Formularios de Servicios · Etapa 1 (Desarrollo)

Enunciado

COMO

usuario empresario que recibe una recomendación de servicio y decide inscribirse.

QUIERO

ser llevado directamente al formulario correcto correspondiente al servicio, con mis datos de contexto disponibles.

PARA

completar mi inscripción sin fricciones ni búsquedas adicionales en el portal.

1. Criterios de Aceptación E1 – Desarrollo
  1. Enlace correcto por servicio (catálogo → formulario)
    • Cada servicio del catálogo tiene exactamente un enlace de formulario canónico mapeado; el mapeo reside en el Catálogo de Servicios (no hardcodeado en el frontend).
    • El sistema genera un deep link con parámetros de contexto: service_id, recommendation_id, user_key_pseudo, canal, campaign_id (si aplica).
    • La redirección se produce en ≤ 500 ms desde el clic del usuario.
  2. Validación automática y monitoreo de enlaces
    • Job automático (cron diario) valida que todos los enlaces de formulario en el catálogo responden con HTTP 200; los que fallan se marcan como enlace_roto.
    • Alerta a equipo de integración cuando se detecta un enlace_roto.
    • Métricas de monitoreo: tasa_enlaces_rotos, latencia_redirect_p95.
  3. Rendimiento y experiencia de navegación
    • La apertura del formulario se produce en una nueva pestaña (o navegación configurada por canal) para no interrumpir el contexto del recomendador.
    • Si la sesión del usuario está activa en el portal CCB, el formulario no requiere re-autenticación (SSO/token heredado).
    • El formulario destino recibe los parámetros de contexto vía query string firmado o JWT corto.
  4. Fallback si el enlace no existe o falla
    • Si el enlace del formulario no existe en el catálogo o está marcado como enlace_roto, se muestra al usuario un mensaje con alternativa de contacto (teléfono o chat CCB).
    • El sistema registra el intento fallido en log con: service_id, motivo_fallo, timestamp.
    • Nunca se redirige al usuario a una URL inválida o de error 404.
  5. Seguridad y control de acceso
    • Los deep links son firmados (HMAC o JWT de corta vida, TTL ≤ 15 minutos) para prevenir manipulación de parámetros.
    • No se expone PII del usuario en la URL; solo user_key_pseudo y el token firmado.
    • El formulario destino valida la firma antes de usar los parámetros de contexto.
  6. Eventos analíticos estructurados
    • Se emite evento recommendation_cta_clicked al hacer clic en el botón de inscripción.
    • Se emite evento form_redirect_initiated con resultado: success / fallback / error.
    • Cada evento incluye: recommendation_id, service_id, user_key_pseudo, canal, campaign_id, timestamp.
2. Tecnología Propuesto E1
Catálogo de Servicios (fuente de mapeo)
  • Campo url_formulario en el maestro de servicios; editable por equipo de integración sin redeploy.
  • Campo estado_enlace: activo / enlace_roto / sin_formulario.
  • API interna para consulta: GET /catalog/services/{id}/form-url.
Servicio de Deep Links
  • Generación de JWT firmado con contexto: service_id, recommendation_id, user_key_pseudo, exp (15 min).
  • Endpoint: POST /deep-links → retorna URL firmada.
  • Integración con SSO del portal para herencia de sesión.
Monitoreo de Enlace
  • Job cron diario: HEAD request a cada URL de formulario del catálogo.
  • Actualización automática del campo estado_enlace.
  • Alerta por email/Teams cuando se detecta enlace_roto.
Pipeline Analítico
  • Eventos estructurados emitidos a pipeline de analítica (Kafka / Event Hub o similar).
  • Registro de: clic en CTA, resultado de redirección, fallback activado.
  • Trazabilidad de recommendation_id → clic → formulario.
3. Flujo Técnico y de Datos E1 – Desarrollo
ACTOR

Usuario empresario que recibe una recomendación de servicio y hace clic en el botón de inscripción.

PARTICIPANTES
  • Usuario / Actor
  • Motor / API (recomendador + servicio deep links)
  • Catálogo Servicios (mapeo service → form URL)
  • Frontend / UI (portal CCB)
  • Formulario (sistema destino)
CONDICIÓN DE ÉXITO

El usuario llega al formulario correcto con el contexto de la recomendación disponible, en ≤ 500ms, sin reautenticación.

Pasos del Flujo (según Diagrama de Secuencia)
  1. Enlace correcto por servicio: el Usuario hace clic en "Inscribirme"; el Motor/API consulta al Catálogo Servicios la URL canónica del formulario para el service_id.
  2. Validación automática y monitoreo: el Motor/API verifica el estado del enlace en el Catálogo (activo vs enlace_roto); si está activo, genera el deep link firmado.
  3. Rendimiento y experiencia de navegación: el Frontend/UI recibe el deep link y lo abre en nueva pestaña hacia el Formulario destino; la sesión SSO se hereda.
  4. Fallback si el enlace no existe o falla: el Formulario devuelve error o el enlace está marcado como roto; el Motor/API activa el fallback y devuelve respuesta alternativa al Motor.
  5. Seguridad y control de acceso: el Motor/API valida la firma del JWT antes de procesar los parámetros de contexto.
  6. Eventos analíticos estructurados: el Catálogo Servicios emite evento form_redirect_initiated al pipeline analítico.
  7. Confirmación final: el Motor/API confirma al usuario el resultado de la redirección; se emiten los eventos analíticos completos.

4. Diseño — Qué ve el Usuario CTA en tarjeta de recomendación
Tarjeta de Recomendación (CTA)
  • Botón primario: "Inscribirme" o "Ir al formulario".
  • Indicador de carga (spinner) mientras se genera el deep link (≤ 500ms).
  • Si hay fallback: mensaje con opción de contacto alternativa.
Formulario Destino
  • Se abre en nueva pestaña con el contexto de la recomendación disponible.
  • El nombre del servicio puede pre-completarse si el formulario lo soporta.
  • Sin solicitud de re-login si la sesión SSO está activa.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Enlace activo — redirección exitosa Deep link generado en ≤ 500ms; usuario llega al formulario correcto con JWT válido.
2 Enlace roto en catálogo Se muestra mensaje de fallback con alternativa de contacto; no se redirige a URL inválida.
3 JWT expirado (> 15 min) El formulario destino rechaza el JWT; el Motor genera uno nuevo si el usuario reintenta.
4 Parámetros manipulados en URL Validación de firma falla; redirección no se procesa; evento de seguridad registrado.
5 Monitoreo detecta enlace roto Job cron actualiza estado a enlace_roto y dispara alerta al equipo de integración.
6 Eventos analíticos completos Cada clic exitoso genera recommendation_cta_clicked y form_redirect_initiated con todos los campos requeridos.
6. Glosario
Deep link
URL que apunta directamente a una página o recurso específico dentro de una aplicación, con parámetros de contexto incluidos.
JWT (JSON Web Token)
Token compacto y firmado que transporta claims de contexto; su firma permite verificar que no fue manipulado.
SSO (Single Sign-On)
Mecanismo de autenticación única que permite al usuario acceder a múltiples sistemas con una sola sesión activa.
Enlace roto
URL de formulario que no responde con HTTP 200; detectada por el job de monitoreo automático.
recommendation_id
Identificador único de la recomendación específica que originó el clic del usuario; usado para trazabilidad en el pipeline de atribución.
Canal
Punto de contacto del usuario con el recomendador (portal web, móvil, asistente, etc.).
Nota de alcance E1: La redirección con deep link y JWT firmado es el contrato mínimo de integración. El pre-llenado de campos en el formulario destino depende de las capacidades del sistema de formularios CCB y se implementa de forma incremental. Los eventos analíticos generados aquí son la base del pipeline de atribución de HU 2.11.2 y 2.11.3.
