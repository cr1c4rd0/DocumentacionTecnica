Registro del resultado de inscripción

Épica 2.11 · Integración con Formularios de Servicios · Etapa 1 (Desarrollo)

Enunciado

COMO

sistema del Recomendador de Servicios (actor técnico — analítica).

QUIERO

recibir y registrar el estado final de cada intento de inscripción originado en una recomendación.

PARA

medir la tasa de conversión real y atribuir correctamente los resultados al motor de recomendación.

1. Criterios de Aceptación E1 – Desarrollo
  1. Registro de estados estándar por intento
    • El sistema registra cada intento de inscripción con uno de los siguientes estados:
      1. inscrito: el usuario completó y envió el formulario exitosamente.
      2. pendiente: el formulario fue enviado pero requiere revisión/confirmación externa.
      3. rechazado: la inscripción fue rechazada por el sistema de formularios (elegibilidad, cupo, etc.).
      4. abandonado: el usuario abrió el formulario pero no lo completó (ver criterio 6).
    • Cada registro incluye obligatoriamente: recommendation_id, service_id, user_key_pseudo, estado, timestamp_intento, timestamp_resultado.
  2. Disponibilidad analítica y trazabilidad
    • Los resultados de inscripción están disponibles en el pipeline analítico en ≤ 5 minutos tras registrarse el evento.
    • Se puede trazar el camino completo: recomendación → clic CTA → apertura formulario → resultado inscripción.
    • Dashboard básico muestra tasa de conversión por servicio, canal y período.
  3. Fuente de verdad del resultado
    • El estado definitivo de una inscripción proviene del sistema de formularios (Formulario) vía webhook/callback; el motor de recomendación es receptor, no origen.
    • Si el formulario no envía callback en ≤ 1 hora, se ejecuta reconciliación automática: el sistema consulta al Formulario el estado del intento por recommendation_id.
    • El estado más reciente del formulario sobrescribe cualquier estado previo; se mantiene historial de cambios.
  4. Captura de detalle de error (mínimo viable)
    • Para inscripciones con estado rechazado, se captura el código de error del formulario (p. ej. cupo_agotado, no_elegible, datos_invalidos).
    • El código de error se almacena en el registro de inscripción; no se expone al usuario directamente (solo se usa para analítica interna).
  5. Rendimiento y resiliencia (no impactar UX)
    • El registro del resultado es asíncrono: un fallo en el registro no interrumpe la experiencia del usuario en el formulario.
    • Reintentos automáticos con backoff exponencial (máx. 3 reintentos) si el endpoint de registro no está disponible.
    • Cola de mensajes con dead letter queue para eventos que no se procesan tras los reintentos.
    • Latencia del endpoint de recepción de callback: p95 ≤ 200ms.
  6. Definición operativa de "abandono"
    • Una inscripción se marca como abandonado si:
      • Han transcurrido ≥ 2 horas desde la apertura del formulario, y
      • No se ha recibido callback de resultado (inscrito, pendiente o rechazado).
    • La ventana de 2 horas es configurable sin redeploy (tabla de configuración).
    • El estado abandonado puede ser sobrescrito si llega un callback tardío con resultado real.
2. Tecnología Propuesto E1
Endpoint de Recepción de Callbacks
  • POST /inscriptions/callback — recibe resultado del formulario.
  • Validación de autenticidad: HMAC-SHA256 con secret compartido por servicio.
  • Idempotente: reintentos del formulario no generan duplicados (recommendation_id como clave idempotente).
SIREP / Almacenamiento
  • Tabla de inscripciones en SIREP o base analítica con campos obligatorios y campo error_code nullable.
  • Índice por recommendation_id, service_id, user_key_pseudo y estado.
  • Historial de cambios de estado (tabla de auditoría con timestamp por cada transición).
Resiliencia y Cola
  • Cola de mensajes (Azure Service Bus / Kafka) para procesamiento asíncrono de callbacks.
  • Política de reintentos: 3 intentos con backoff 1s → 5s → 30s.
  • Dead letter queue para eventos sin procesar; alerta al equipo de integración.
Reconciliación y Abandono
  • Job cron cada 30 minutos: identifica intentos sin callback en el tiempo configurado.
  • Para cada intento huérfano: consulta estado al Formulario por recommendation_id.
  • Si no hay respuesta, marca como abandonado según ventana configurada (default 2h).
3. Flujo Técnico y de Datos E1 – Desarrollo
ACTOR

Sistema del Recomendador (actor técnico) — recibe y registra resultados de inscripción para analítica.

PARTICIPANTES
  • Usuario / Actor
  • Motor / API (endpoint callback + analítica)
  • SIREP (almacenamiento resultados)
  • Frontend / UI (visualización tasa conversión)
  • Formulario (fuente de verdad del resultado)
ESTADOS POSIBLES
inscrito pendiente rechazado abandonado
Pasos del Flujo (según Diagrama de Secuencia)
  1. Registro de estados estándar por intento: el Motor/API abre el registro de inscripción con estado pendiente al recibir el evento de apertura de formulario.
  2. Disponibilidad analítica y trazabilidad: el Motor/API envía el registro a SIREP para disponibilidad analítica; SIREP confirma almacenamiento.
  3. Fuente de verdad del resultado: el Formulario (sistema destino) envía callback con estado definitivo al Motor/API; el estado se actualiza en SIREP.
  4. Captura de detalle de error: el Formulario incluye código de error en el callback cuando aplica (estado rechazado); el Motor almacena el error_code en SIREP.
  5. Rendimiento y resiliencia: el Motor/API confirma recepción del callback a SIREP y emite evento analítico; la operación es asíncrona y no bloquea al usuario.
  6. Definición operativa de abandono: el Motor devuelve la definición operativa de abandono al usuario/analítica y cierra el ciclo del intento.

4. Diseño — Dashboard de Conversiones Vista analítica interna
KPIs Principales
  • Tasa de conversión global: inscripciones / clics en CTA.
  • Desglose por servicio, canal y período.
  • Distribución de estados: inscrito / pendiente / rechazado / abandonado.
Alertas y Monitoreo
  • Alerta cuando la tasa de conversión de un servicio cae por debajo de umbral configurado.
  • Alerta cuando hay acumulación en dead letter queue.
  • Alerta cuando la reconciliación automática detecta alta tasa de intentos huérfanos.
5. Casos de Prueba Mínimos E1
# Escenario Resultado Esperado
1 Callback exitoso — inscrito Registro actualizado a inscrito en ≤ 200ms; disponible en analítica en ≤ 5min.
2 Callback con rechazo + código error Registro actualizado a rechazado; campo error_code almacenado; no se expone al usuario.
3 Sin callback en 2 horas Job de reconciliación marca registro como abandonado; log registra motivo.
4 Callback tardío (después de marcado como abandonado) Estado se actualiza al valor real del callback; historial de cambios conserva el estado previo.
5 Callback duplicado (reintento del formulario) Endpoint idempotente: no se genera registro duplicado; se retorna ok.
6 Endpoint no disponible Cola retiene el evento; reintentos con backoff; tras 3 fallos, evento va a dead letter queue.
7 Trazabilidad completa Dado un recommendation_id, se puede trazar: recomendación → clic → apertura formulario → resultado.
6. Glosario
Callback / Webhook
Llamada HTTP que el sistema de formularios hace al Recomendador para notificar el resultado de una inscripción.
Idempotencia
Propiedad de una operación que produce el mismo resultado aunque se ejecute múltiples veces con los mismos parámetros.
Dead letter queue
Cola especial que recibe los mensajes que no pudieron procesarse tras el número máximo de reintentos, para revisión manual.
Reconciliación
Proceso automático que detecta y resuelve discrepancias entre el estado esperado y el estado real de un registro.
Tasa de conversión
Proporción de clics en CTA que resultan en una inscripción exitosa (inscrito).
HMAC
Hash-based Message Authentication Code — mecanismo criptográfico para verificar la autenticidad de un mensaje.
Nota de alcance E1: El registro del resultado de inscripción es la base del pipeline de atribución. En E1 se capturan todos los estados estándar y se garantiza trazabilidad por recommendation_id. La lógica de atribución multi-toque y ventanas de conversión parametrizables se especifica en HU 2.11.3.
