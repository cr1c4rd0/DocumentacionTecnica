Exclusión de servicios ya consumidos

Épica 2.7 · Reglas de Exclusión · No recomendar "lo que ya hice"

Enunciado

COMO

usuario (empresario o ciudadano) que recibe recomendaciones de servicios CCB.

QUIERO

que el motor no me recomiende servicios que ya he tomado (contratado, inscrito o completado), con posibilidad de configurar excepciones para servicios renovables.

PARA

ver solo opciones nuevas y relevantes para mi situación actual, sin ruido de servicios ya aprovechados.

1. Criterios de Aceptación Reglas de Exclusión
  1. Identificación de servicios "ya consumidos"
    • El motor consulta el historial del usuario contra la fuente de verdad (SIREP / CRM Dynamics) antes de generar el Top-N.
    • Se consideran "consumidos" los servicios con estado: contratado, inscrito, completado, en_curso.
    • La consulta se realiza por user_key_pseudo para preservar seudonimización en entorno de desarrollo.
  2. Exclusión del Top-N recomendado por defecto
    • Todo servicio con estado "consumido" queda excluido del conjunto candidato antes del scoring.
    • El motivo de exclusión se registra como reason: "taken" en el log de decisiones.
    • El Top-N resultante no contiene ningún ítem presente en services_taken[].
  3. Manejo de excepciones y casos especiales
    • Servicios renovables: aquellos marcados con renovable: true en el catálogo pueden volver a recomendarse tras un período de "cooling-off" configurable (p. ej., 6 meses).
    • Override administrativo: un operador puede marcar manualmente un servicio como "re-recomendable" para un usuario específico, registrando justificación y timestamp.
    • Servicio actualizado: si un servicio tiene una nueva edición/cohorte, se trata como ítem diferente y puede recomendarse aunque exista historial de la versión anterior.
  4. Trazabilidad
    • El log de cada request incluye: user_key_pseudo, services_excluded[] (con motivo taken), version_regla, timestamp.
    • Se puede reproducir qué servicios fueron excluidos para un usuario y momento dados.
  5. Experiencia UX coherente
    • La lista de recomendaciones mostrada al usuario es "limpia": solo contiene ítems no tomados previamente.
    • Si todos los servicios elegibles ya fueron tomados, se muestra un mensaje contextual: "Has aprovechado todo lo disponible para tu perfil. Pronto habrá nuevas opciones."
    • El usuario puede acceder a un panel de "mis servicios tomados" para referencia propia (control del usuario).
2. Tecnología Stack Propuesto
Fuente de Historial
  • SIREP (Sistema de Registro de Servicios): fuente primaria de services_taken[].
  • CRM Dynamics: fuente secundaria para validar estado del cliente y registros históricos.
  • En E1 (dev): extracto seudonimizado o mock con datos representativos.
Filtro de Exclusión
  • Aplicado en la fase de candidatos del pipeline, antes del scoring.
  • Join entre conjunto candidato y tabla user_service_history indexada por (user_key_pseudo, service_id).
  • TTL de exclusión configurable por categoría de servicio (YAML/tabla de configuración).
Logging y Auditoría
  • Logs estructurados en JSON con campo exclusion_reasons[].
  • Almacenamiento en Data Lake / tabla de auditoría con retención configurable.
  • Dashboard de monitoreo: % de candidatos excluidos por motivo taken por sesión.
Configuración de Excepciones
  • Tabla service_exclusion_config: service_id, renovable, cooling_off_days, override_allowed.
  • Cambios auditados con versión y timestamp; hot-reload sin redeploy del modelo.
3. Flujo Técnico y de Datos Pipeline de Exclusión
ACTORES

Usuario / Actor (empresario o ciudadano) · Motor / API · CRM / Dynamics · Frontend / UI

DISPARADOR

Solicitud de recomendaciones (on-demand o proactiva). El sistema identifica los servicios "ya consumidos" por el usuario antes de generar el Top-N.

RESULTADO

Lista Top-N "limpia" sin servicios ya tomados, con log de exclusiones trazable y mensaje contextual cuando aplique.

Pasos del Flujo (según Diagrama de Flujo y Secuencia)
  1. Identificar servicios "ya consumidos": el Motor / API consulta SIREP y CRM para obtener services_taken[] del usuario.
  2. Excluir del Top-N: CRM / Dynamics confirma estados; el motor filtra los candidatos eliminando todos los servicios en services_taken[].
  3. Manejar excepciones y casos especiales: el Frontend / UI envía señales de servicios renovables u overrides; el motor aplica las reglas de excepción configuradas.
  4. Mantener trazabilidad: el motor registra en CRM / Dynamics cada decisión de exclusión con motivo, versión de regla y timestamp.
  5. Garantizar experiencia UX coherente: CRM envía al Frontend la lista limpia y los metadatos de contexto (mensaje si lista vacía).
  6. Recomendaciones "limpias": el Frontend / UI recibe y renderiza el Top-N filtrado.
  7. Mensaje contextual (cuando aplique): el Motor envía al Frontend el mensaje adecuado si todos los elegibles fueron tomados.
  8. Control del usuario: el usuario confirma recepción; puede acceder a panel de "mis servicios tomados".

4. Diseño — Experiencia del Usuario
Lista de Recomendaciones (estado normal)
  • Top-N renderizado sin servicios ya tomados.
  • Cada tarjeta muestra nombre, descripción breve, razón de recomendación y CTA.
  • Badge Nuevo para ti en servicios nunca visitados.
Estado: Sin recomendaciones nuevas
  • Mensaje contextual: "Has aprovechado todo lo disponible para tu perfil. Pronto habrá nuevas opciones."
  • Enlace a "Mis servicios tomados" para que el usuario revise su historial.
  • Sugerencia de contacto con asesor si necesita orientación adicional.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Servicio en services_taken[] El servicio S-200 no aparece en el Top-N; log registra reason: "taken".
2 Todos los elegibles tomados Top-N retorna lista vacía; UI muestra mensaje contextual de "todo aprovechado".
3 Servicio renovable fuera de cooling-off Servicio con renovable: true y cooling_off expirado reaparece en candidatos.
4 Override administrativo Operador activa override; el servicio es incluido en candidatos y el log registra reason: "admin_override".
5 Reproducibilidad del log Dado el request_id, se puede reconstruir qué servicios fueron excluidos y por qué.
6 Nueva edición de servicio tomado Servicio con nueva edición/cohorte (service_id diferente) se recomienda aunque exista historial de la versión anterior.
6. Glosario
services_taken[]
Lista de IDs de servicios que el usuario ya ha contratado, inscrito o completado según el historial.
reason: "taken"
Código de motivo de exclusión registrado en el log cuando un servicio se descarta por haber sido previamente consumido.
Cooling-off
Período mínimo de tiempo que debe transcurrir antes de que un servicio renovable pueda volver a recomendarse.
Override administrativo
Acción manual de un operador para permitir la recomendación de un servicio normalmente excluido, con justificación auditada.
Top-N limpio
Lista de recomendaciones filtrada donde ningún ítem pertenece al historial de consumo del usuario.
user_key_pseudo
Identificador seudonimizado del usuario, no reversible, utilizado en entorno de desarrollo para preservar privacidad.
Nota de alcance: Esta regla de exclusión opera en la fase de generación de candidatos (pre-scoring), garantizando que el motor nunca puntúe ni devuelva servicios ya consumidos. Las excepciones (renovables, overrides) están controladas por configuración versionada sin redeploy.
