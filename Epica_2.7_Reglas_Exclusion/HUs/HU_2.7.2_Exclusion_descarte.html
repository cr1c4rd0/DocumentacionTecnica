Exclusión por descarte explícito (hide/dismiss)

Épica 2.7 · Reglas de Exclusión · Ventana temporal y control del usuario

Enunciado

COMO

usuario (empresario o ciudadano) que recibe recomendaciones del Recomendador de Servicios CCB.

QUIERO

poder descartar explícitamente una recomendación (acción hide/dismiss) para que no vuelva a aparecer durante una ventana temporal configurable, con opción de descarte permanente y posibilidad de revertir.

PARA

tener control real sobre mi experiencia de recomendaciones y no ver repetidamente servicios que rechacé conscientemente.

1. Criterios de Aceptación Descarte Explícito
  1. Descarte desde la lista recomendada (acción hide/dismiss)
    • Cada tarjeta de recomendación expone un botón / ícono "No me interesa" (hide/dismiss).
    • Al accionarlo, la tarjeta desaparece de la lista inmediatamente (feedback visual instantáneo).
    • Se muestra un toast de confirmación: "Ocultamos esta recomendación. ¿Deshacer?" con ventana de 5 s para revertir.
    • Opcionalmente, se invita al usuario a proveer una razón no intrusiva: "No es para mí ahora", "Ya lo conozco", "Otro".
  2. Lista de exclusión con ventana temporal
    • El servicio descartado pasa a una lista de exclusión dismissed_services[] con: service_id, user_key_pseudo, dismissed_at, window_days, permanent, reason_opt.
    • La ventana de exclusión predeterminada es de 30 días, configurable por tipo de servicio o por admin.
    • Transcurrida la ventana, el servicio vuelve al pool de candidatos elegibles automáticamente.
  3. El motor no vuelve a mostrar el servicio descartado (durante la ventana)
    • En cada solicitud de recomendaciones, el motor consulta dismissed_services[] y excluye los ítems vigentes.
    • El motivo de exclusión se registra como reason: "dismissed" en el log de decisiones.
    • El servicio descartado no aparece en ningún canal mientras la ventana esté activa.
  4. Habilitar control y reversibilidad
    • El usuario puede acceder a un panel "Mis preferencias / Descartados" donde ve la lista de servicios ocultos, su ventana activa y fecha de expiración.
    • Puede restaurar un servicio descartado manualmente (antes de que expire la ventana).
    • Puede marcar un servicio como "No mostrar nunca" (permanent: true), eliminando la ventana temporal.
  5. Registrar eventos y medir impacto
    • Cada acción de descarte genera un evento: event_type: "dismiss", service_id, reason_opt, timestamp, canal.
    • Métricas disponibles: tasa de descarte por servicio, distribución de razones, porcentaje de reversiones, impacto en cobertura del motor.
    • En caso de error al registrar el descarte (p. ej., fallo de red), se notifica al usuario y se reintenta en background; la tarjeta no reaparece en la sesión activa.
2. Tecnología Stack Propuesto
Almacenamiento de Descartes
  • Tabla user_dismissed_services con campos: user_key_pseudo, service_id, dismissed_at, window_days, permanent, reason_opt, restored_at.
  • Índice optimizado por (user_key_pseudo, service_id, expires_at).
  • TTL gestionado por job nocturno o partición por fecha de expiración.
API de Descarte
  • POST /recommendations/{service_id}/dismiss — registra el descarte.
  • DELETE /dismissed/{service_id} — restaura el servicio.
  • GET /dismissed — lista los servicios descartados del usuario con estado y expiración.
  • Respuesta sincrónica con confirmación; latencia objetivo < 200 ms.
Integración con el Motor
  • El filtro de descartes se aplica en la fase de candidatos, junto al filtro taken.
  • La configuración de ventanas temporales por categoría se gestiona en YAML/tabla versionada.
  • Sincronización entre canales (web, app, chat) mediante store compartido.
UX y Notificaciones
  • Toast de confirmación con acción "Deshacer" (5 s).
  • Micro-interacción: la tarjeta se desliza fuera de la vista (animación CSS).
  • Panel "Mis preferencias": lista paginada con CTA de restaurar / hacer permanente.
3. Flujo Técnico y de Datos Secuencia de Descarte
ACTORES

Usuario / Actor · Motor / API · Frontend / UI · Asistente / Chat

DISPARADOR

El usuario hace clic en "No me interesa" en una tarjeta de recomendación, desde cualquier canal (web, app, chat).

RESULTADO

Servicio excluido de futuras recomendaciones durante la ventana. Evento registrado. Control reversible en manos del usuario.

Pasos del Flujo (según Diagrama de Flujo y Secuencia)
  1. Descarte desde la lista recomendada: el usuario acciona el botón hide/dismiss en una tarjeta; el Motor / API recibe el evento.
  2. El servicio pasa a lista de exclusión: el Frontend / UI actualiza el estado local; el Motor persiste el registro en dismissed_services[].
  3. El motor no vuelve a mostrar ese servicio: en futuras solicitudes, el Motor excluye el ítem; el Frontend no lo renderiza.
  4. Habilitar control y reversibilidad: el Motor expone el endpoint de restauración; el Frontend muestra el panel de gestión de descartados.
  5. Registrar eventos y medir impacto: el Frontend envía el evento de descarte al Asistente / Chat y a la capa de analytics.
  6. En cada tarjeta de recomendación: el Asistente / Chat confirma al Motor los eventos registrados.
  7. Toast de confirmación: el Motor envía al Frontend el mensaje de confirmación con opción "Deshacer" (5 s).
  8. Razón opcional (no intrusiva): el usuario puede seleccionar una razón; el Motor la registra en reason_opt.
Ejemplo de Registro de Descarte
{
  "event_type": "dismiss",
  "user_key_pseudo": "u_9f3a",
  "service_id": "S-405",
  "dismissed_at": "2025-03-10T15:10:00Z",
  "window_days": 30,
  "permanent": false,
  "reason_opt": "No es para mí ahora",
  "canal": "web",
  "version_regla": "exclusion-rules-2025.03.01"

4. Diseño — Experiencia del Usuario
Tarjeta de Recomendación con Descarte
  • Ícono "✕" o menú "..." en esquina superior derecha de la tarjeta.
  • Al hacer clic: animación de deslizamiento + toast de confirmación.
  • Toast: "Ocultamos esta recomendación" + botón "Deshacer" (5 s).
  • Opción secundaria (no modal): selector de razón con máx. 3 opciones simples.
Panel "Mis preferencias / Descartados"
  • Lista de servicios descartados con nombre, fecha de descarte y fecha de expiración.
  • Botón "Restaurar" por ítem para reactivar antes de que expire la ventana.
  • Toggle "No mostrar nunca" para hacer el descarte permanente.
  • Accesible desde menú de usuario / configuración de preferencias.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Descarte básico Al descartar S-405, la tarjeta desaparece de inmediato y el evento se registra con reason: "dismissed".
2 Ventana temporal activa S-405 no aparece en el Top-N durante 30 días; reaparece automáticamente el día 31.
3 Deshacer (Undo) El usuario hace clic en "Deshacer" dentro de 5 s; la tarjeta reaparece y el registro de descarte se elimina.
4 Descarte permanente Usuario activa "No mostrar nunca"; permanent: true persiste; el servicio no reaparece tras ningún período.
5 Restaurar desde panel Usuario restaura S-405 desde el panel; en la siguiente sesión el servicio puede volver a recomendarse.
6 Error de registro (fallback) Si falla la persistencia, se notifica al usuario; la tarjeta no reaparece en la sesión activa y se reintenta en background.
7 Sincronización entre canales Servicio descartado en web no aparece en app ni en chat durante la ventana activa.
6. Glosario
hide/dismiss
Acción explícita del usuario para ocultar una recomendación de su lista activa.
dismissed_services[]
Tabla/lista que almacena los servicios descartados por el usuario con metadatos de ventana temporal.
window_days
Número de días durante los cuales el servicio descartado permanece excluido del pool de candidatos.
permanent: true
Flag que indica que el descarte no expira; el servicio nunca volverá a recomendarse al usuario.
reason_opt
Razón opcional (no obligatoria) que el usuario puede seleccionar al descartar, usada para mejorar el modelo.
Toast
Notificación emergente breve (no modal) que confirma una acción y ofrece la opción de revertirla.
Nota de alcance: El descarte explícito opera complementariamente al filtro de servicios "ya tomados" (HU 2.7.1). Ambas reglas se aplican en la fase de candidatos. La ventana temporal predeterminada (30 días) y el umbral de descarte permanente son configurables vía tabla de configuración versionada, sin redeploy del modelo.
