Historia de Usuario · HU 2.8.1

UI de carruseles para explorar recomendaciones

Épica 2.8 · UI Recomendador tipo Catálogo · Catálogo navegable horizontal

Enunciado

COMO

usuario (empresario o ciudadano) que explora servicios en el Recomendador CCB.

QUIERO

explorar mis recomendaciones organizadas en carruseles horizontales por sección temática, con tarjetas de servicio que muestran información clave y estados del motor integrados.

PARA

descubrir servicios de forma visual e intuitiva, con un catálogo navegable que no me abrume y que refleje en tiempo real el estado del motor de recomendación.

1. Criterios de Aceptación UI Carruseles
  1. Organizar recomendaciones en carruseles (secciones horizontales)
    • Las recomendaciones se agrupan en secciones temáticas (p. ej., Formalización, Legal, Financiero, Formación, Internacional) con título visible por sección.
    • Cada sección es un carrusel de scroll horizontal con flechas de navegación (< / >) y soporte de swipe en móvil.
    • El número mínimo de secciones visibles es 3; el máximo configurable por admin.
  2. Tarjetas de servicio (cards) con información esencial
    • Cada tarjeta expone: nombre del servicio, descripción breve (máx. 2 líneas), modalidad (presencial/virtual), duración estimada, costo (gratuito / con cargo), badge de razón de recomendación.
    • Badges contextuales: Nuevo, Popular en tu segmento, Campaña activa, Recomendado por tu perfil.
    • CTA primario: botón "Ver detalle" que lleva al detalle del servicio o al formulario de inscripción.
    • CTA secundario: ícono de descarte (hide/dismiss, integrado con HU 2.7.2).
  3. Integración de estados del motor
    • Estado de carga: skeleton screens (placeholders animados) mientras el API responde.
    • Estado vacío: mensaje contextual si una sección no tiene recomendaciones disponibles: "No hay recomendaciones nuevas en esta área por ahora."
    • Estado de error: fallback visual con botón "Reintentar"; se muestra mensaje no técnico al usuario.
    • Estado actualizado: indicador visual cuando el motor actualiza las recomendaciones en tiempo real (badge "Actualizado").
  4. Prototipo UX validable y versión productiva
    • Se entrega un prototipo navegable (Figma / HTML estático) con datos mock para validación con usuarios.
    • La versión productiva consume el API del recomendador con datos reales (autenticado por user_key_pseudo).
    • Ambas versiones son accesibles: WCAG 2.1 AA, navegables por teclado, compatibles con lectores de pantalla.
    • Diseño responsive: 1 tarjeta en mobile, 2 en tablet, 3–4 en desktop.
  5. Consumo de API del recomendador
    • La UI consume el endpoint GET /recommendations?user_key_pseudo=&k=&canal=web.
    • Las tarjetas se renderizan con los campos: service_id, nombre, score, razones[], estado_disponibilidad.
    • Lazy loading: se cargan máx. 10 tarjetas por sección; "Ver más" carga el siguiente lote.
  6. Interacción y CTAs
    • Cada tarjeta permite "Ver detalle" (modal o nueva vista) con descripción completa, razones expandidas y acceso al formulario.
    • Las interacciones (clic en tarjeta, CTA, dismiss) generan eventos de telemetría para el motor de aprendizaje continuo.
2. Tecnología Stack Propuesto
Frontend — Componente Carrusel
  • Web Component reutilizable (o React component) <recommendation-carousel>.
  • Scroll horizontal nativo con CSS overflow-x: auto + JavaScript para flechas.
  • Intersection Observer para lazy load y prefetch del siguiente lote.
  • Soporte de swipe táctil (Touch Events / Pointer API).
API y Datos
  • Endpoint REST: GET /recommendations con parámetros user_key_pseudo, k, canal, section.
  • Respuesta tipada con service_id, nombre, score, razones[], modalidad, duracion, costo, badges[].
  • Caché client-side (SWR / React Query) con revalidación en foco.
Accesibilidad y Rendimiento
  • WCAG 2.1 AA: roles ARIA (role="region", aria-label), foco en flechas y tarjetas.
  • Skeleton screens con aria-busy="true" durante carga.
  • Core Web Vitals: LCP < 2.5 s, CLS < 0.1.
  • Imágenes de servicio: lazy loading con loading="lazy".
Telemetría e Integración
  • Eventos de interacción: recommendation_viewed, recommendation_clicked, cta_clicked, recommendation_dismissed.
  • Integración con HU 2.7.2 (dismiss) y HU 2.9.1 (explicación por recomendación).
  • Logs estructurados enviados al backend de analytics.
3. Flujo Técnico y de Datos Secuencia UI-Motor
ACTORES

Usuario / Actor · Motor / API · Frontend / UI

DISPARADOR

El usuario accede a la sección de recomendaciones del portal CCB (o el componente se monta en cualquier página que lo incluya).

RESULTADO

Carruseles renderizados con tarjetas de servicio personalizadas, estados de carga/error integrados y telemetría de interacción activa.

Pasos del Flujo (según Diagramas)
  1. Organizar recomendaciones en carruseles: el Motor / API recibe la solicitud del usuario y devuelve recomendaciones agrupadas por sección temática.
  2. Presentar tarjetas de servicio: el Frontend / UI renderiza las tarjetas con la información del API (nombre, score, razones, badges, CTAs).
  3. Integrar estados del motor: el Motor / API informa el estado de cada sección (cargando, vacía, error, actualizada); el Frontend adapta el renderizado.
  4. Entregar prototipo UX y versión productiva: el Frontend / UI valida el prototipo con datos mock; en producción consume el API real.
  5. Verificar carruseles funcionales en prototipo: el Motor / API confirma que el prototipo puede consumir el contrato de API definido.
  6. Verificar carruseles funcionales en productivo: el Frontend / UI valida el contrato de API con datos reales.
  7. Consumo de API del recomendador: el Motor / API responde con el Top-N estructurado; el Frontend renderiza el carrusel completo.
  8. Interacción y CTAs: el usuario interactúa (clic en tarjeta, "Ver detalle", dismiss); los eventos se envían al Motor para telemetría.

4. Diseño — Qué ve el Usuario
Prototipo (validación con datos mock)
  • Secciones temáticas con título y contador de ítems.
  • Tarjetas con nombre, descripción (2 líneas), modalidad, duración y CTA.
  • Flechas de navegación y soporte swipe.
  • Skeleton screens visibles durante carga simulada.
Versión Productiva
  • Datos reales del Motor de Recomendación; actualización en foco o por intervalo.
  • Badges dinámicos generados por el motor (razones[]).
  • Integración con dismiss (HU 2.7.2) y explicación (HU 2.9.1).
  • Accesible y responsive para todos los dispositivos CCB.
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Carga exitosa de carruseles Se muestran ≥ 3 secciones temáticas con tarjetas; skeleton desaparece al completarse la carga.
2 Navegación por carrusel Las flechas avanzan/retroceden entre tarjetas; el swipe funciona en móvil sin desplazar la página.
3 Estado vacío por sección Sección sin recomendaciones muestra mensaje contextual, no error técnico.
4 Error de API Se muestra fallback visual con "Reintentar"; no se bloquea el resto de secciones.
5 Lazy loading Al llegar al final del carrusel, se carga el siguiente lote de tarjetas sin recargar la página.
6 Accesibilidad Carrusel navegable por teclado (Tab/Enter/flechas); roles ARIA correctos; sin errores en auditoría Axe.
7 Telemetría de interacción Al hacer clic en "Ver detalle", el evento recommendation_clicked se registra con service_id y section_id.
6. Glosario
Carrusel
Componente UI de scroll horizontal que presenta múltiples tarjetas de servicio en una misma sección temática.
Skeleton screen
Placeholder animado que representa la estructura de una tarjeta mientras se cargan los datos reales del API.
Badge
Etiqueta visual en la tarjeta que comunica el contexto de la recomendación (p. ej., "Popular", "Campaña activa").
CTA
Call-to-Action: botón de acción principal o secundario en la tarjeta ("Ver detalle", dismiss).
Lazy loading
Técnica que carga contenido adicional solo cuando el usuario lo requiere (scroll hasta el final del carrusel).
WCAG 2.1 AA
Nivel de conformidad de accesibilidad web que garantiza usabilidad para personas con discapacidades visuales, motrices y cognitivas.
Nota de alcance: Esta HU cubre el componente de carrusel como presentación del catálogo navegable. La lógica de filtros y el botón "Sorpréndeme" se especifican en HU 2.8.2. La integración con dismiss (HU 2.7.2) y con la explicación por recomendación (HU 2.9.1) se realiza mediante los endpoints definidos en dichas HUs.
