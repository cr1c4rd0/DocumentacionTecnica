Épica 2.13 · Catálogo y Promoción de Servicios

Enunciado

COMO

usuario que consulta el Recomendador de Servicios e identifica un servicio de su interés con estado "Sin programación actual" o "Próximamente"

QUIERO

suscribirme para recibir una notificación automática cuando ese servicio abra sus inscripciones o cambie su estado de disponibilidad

PARA

no perder la oportunidad de inscribirme sin tener que revisar el catálogo manualmente, cumpliendo mis preferencias de canal y privacidad

1. Criterios de Aceptación Notificaciones
  1. Suscripción desde servicios "sin programación actual" o "próximamente"
    • El botón "Avisarme cuando abra" aparece únicamente en servicios con estado unscheduled o upcoming.
    • El usuario puede suscribirse con un solo clic si tiene sesión activa; si es anónimo, se le solicita email antes de confirmar.
    • Confirmación visual inmediata con feedback (toast o badge "Suscrito").
    • Un usuario no puede quedar suscrito dos veces al mismo servicio (deduplicación en backend).
  2. Disparo automático de notificación por cambio de estado
    • Cuando el estado de un servicio cambia a available, se dispara automáticamente la notificación a todos los suscriptores en ≤ 1 hora.
    • El disparador se basa en el webhook de cambio de estado del catálogo (no en polling).
    • Se notifica también si el estado cambia a upcoming con fecha definida (para suscripciones desde unscheduled).
  3. Verificación de vigencia semántica
    • Antes de enviar la notificación, el Motor / API verifica que el servicio sigue disponible (no fue cerrado inmediatamente tras abrir).
    • Si la verificación falla (servicio ya cerrado nuevamente), se cancela el envío y se registra el fallback con código de motivo.
    • Si el servicio no existe o fue eliminado del catálogo, se cancela la suscripción y se notifica al usuario con mensaje explicativo.
  4. Gestión de preferencias y cancelación
    • El usuario puede cancelar una suscripción activa desde su perfil o directamente desde el email con un clic (enlace de baja).
    • Las preferencias de canal (email, push, SMS) se respetan según configuración del usuario.
    • La gestión de suscripciones activas es visible desde el perfil (lista de servicios suscritos + canal + fecha de suscripción).
  5. Fallback por falta de canal o datos de contacto
    • Si el usuario no tiene canal de contacto válido configurado, se le informa en el momento de suscripción y se le invita a completar su perfil.
    • Si el envío falla (rebote de email, push no entregado), se registra el error y se aplica retry con backoff exponencial (máx. 3 intentos).
    • Si todos los intentos fallan, se registra notification_failed con motivo y se genera alerta interna.
  6. Cumplimiento y privacidad
    • Solo se envían notificaciones a usuarios con consentimiento explícito de comunicaciones (Ley 1581 / GDPR).
    • El enlace de baja en el email es de un clic y efectivo de forma inmediata.
    • No se persiste PII en logs de notificación; se usa user_key_pseudo o email hasheado para trazabilidad.
    • Rate limiting: máximo N notificaciones por usuario por día (configurable, defecto 3).
2. Tecnología Stack propuesto
Servicio de Suscripciones
  • Tabla de suscripciones: user_key_pseudo, service_id, canal, timestamp_suscripcion, activa.
  • Índice único en (user_key_pseudo, service_id) para deduplicación.
  • API REST: POST /subscriptions, DELETE /subscriptions/{id}, GET /subscriptions.
Disparador y Notificación
  • Trigger en webhook de cambio de estado del Catálogo Servicios.
  • Cola de mensajes (Kafka / SQS) para procesamiento asíncrono de notificaciones.
  • Integración con servicio de emails transaccionales (SendGrid, Amazon SES o similar).
  • Push notifications vía FCM / APNs según preferencias del usuario.
Motor / API y Verificación
  • Verificación de vigencia semántica antes del envío: consulta estado actual del catálogo.
  • Lógica de retry con backoff exponencial (máx. 3 intentos).
  • Rate limiting configurable por usuario y por servicio.
Cumplimiento y Observabilidad
  • Integración con CRM / Dynamics para verificar flag de consentimiento de comunicaciones.
  • Log de notificaciones: service_id, user_key_pseudo, canal, estado_envio, timestamp.
  • Dashboard de suscripciones activas, tasa de entrega y fallbacks.
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: consulta el recomendador y se suscribe.
  • Motor / API: gestiona suscripciones y dispara notificaciones.
  • CRM / Dynamics: disparo automático por cambio de estado.
  • SIREP: verificación de vigencia semántica.
  • Catálogo Servicios: gestión de preferencias y cancelación.
  • Frontend / UI: fallback por falta de canal o datos.
HAPPY PATH
  1. Usuario se suscribe desde servicio "sin programación" (Catálogo Servicios → Motor / API).
  2. CRM / Dynamics detecta cambio de estado y dispara notificación automática al Motor / API.
  3. Motor / API verifica vigencia semántica en SIREP.
  4. Catálogo Servicios gestiona preferencias de canal y cancelación en Frontend / UI.
  5. Frontend / UI reporta fallback si falta canal o datos de contacto al Motor / API.
  6. Motor / API confirma cumplimiento y privacidad al Usuario.
ERROR PATH
  • Si verificación de vigencia semántica falla → Error / Fallback: notificar al usuario que el servicio ya no está disponible.
  • Si canal de contacto no disponible → solicitar actualización de perfil.
  • Si envío falla → retry con backoff; si agota intentos → notification_failed + alerta interna.

4. Diseño — Experiencia de Suscripción
En la Tarjeta de Servicio
  • Botón "Avisarme cuando abra" visible en servicios unscheduled / upcoming.
  • Al hacer clic: confirmación inmediata con toast "¡Listo! Te avisaremos cuando abra."
  • El botón cambia a badge "Suscrito ✓" y queda deshabilitado si el usuario ya está suscrito.
En el Perfil del Usuario
  • Sección "Mis alertas de servicios" con lista de suscripciones activas.
  • Cada suscripción muestra: nombre del servicio, canal, fecha de suscripción y botón "Cancelar alerta".
  • Selector de canal preferido (email / push / SMS).
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Suscripción exitosa Usuario se suscribe a servicio unscheduled; aparece confirmación visual y registro en tabla de suscripciones.
2 Deduplicación Segundo intento de suscripción al mismo servicio retorna 409 Already subscribed; no se crea registro duplicado.
3 Disparo automático ≤ 1 h Al cambiar estado a available, todos los suscriptores reciben notificación en ≤ 1 hora.
4 Verificación de vigencia fallida Si servicio se cierra antes del envío, notificación se cancela y se registra notification_cancelled: service_not_available.
5 Baja desde email Clic en enlace de baja cancela suscripción de forma inmediata; no se envían más notificaciones.
6 Sin canal de contacto Al suscribirse sin email/push configurado, se muestra mensaje solicitando completar perfil; no se crea suscripción huérfana.
7 Cumplimiento de consentimiento Usuario sin flag de consentimiento de comunicaciones no recibe notificación aunque esté suscrito.
6. Glosario
Vigencia semántica
Verificación de que el estado de un servicio sigue siendo el esperado en el momento de enviar la notificación.
Suscripción activa
Registro en tabla de suscripciones con flag activa = true para un par (usuario, servicio).
Backoff exponencial
Estrategia de reintento que duplica el tiempo de espera entre intentos fallidos para evitar sobrecarga.
Rate limiting
Límite máximo de notificaciones enviadas a un usuario en un período de tiempo para evitar spam.
Fire-and-forget
Patrón de envío donde el productor no espera confirmación del consumidor; el procesamiento es asíncrono.
Consentimiento explícito
Aceptación documentada del usuario para recibir comunicaciones, requerida por Ley 1581 / GDPR.
Nota de alcance: Esta HU cubre el ciclo completo de suscripción y notificación de disponibilidad futura. La clasificación y etiquetado de estados de disponibilidad se gestiona en HU 2.13.1. El cumplimiento de privacidad aplica Ley 1581 y lineamientos internos CCB.
