Clasificación de servicios

Épica 2.13 · Catálogo y Promoción de Servicios

Enunciado

COMO

usuario que consulta el Recomendador de Servicios

QUIERO

ver cada servicio con etiquetas visibles de categoría y estado de disponibilidad (Disponible, Próximamente, Cerrado, Sin programación)

PARA

entender de un vistazo qué servicios puedo contratar ahora y cuáles debo esperar, con reglas de clasificación consistentes y trazables

1. Criterios de Aceptación Clasificación
  1. Etiquetas visibles por servicio (categoría + estado)
    • Cada servicio muestra de forma visible su categoría (área temática) y su estado de disponibilidad.
    • Los estados válidos son: Disponible, Próximamente, Cerrado, Sin programación actual.
    • Las etiquetas de estado usan colores diferenciados definidos en design tokens (no hardcoded).
    • Cuando aplique, se muestra el contador de cupos disponibles junto al estado.
  2. Reglas de clasificación consistentes y trazables
    • Las reglas de clasificación por categoría y estado se definen en una fuente única de verdad (tabla de configuración o archivo YAML en repo).
    • Cualquier cambio en la clasificación queda auditado (versión + timestamp + autor).
    • El campo status en el catálogo usa un enum estricto; valores no reconocidos se rechazan con error 400.
    • Las categorías son jerárquicas (área > subárea) y visibles en la tarjeta del servicio.
  3. Actualización y frescura de la disponibilidad
    • Los estados se actualizan vía webhook del gestor de servicios en tiempo real, sin recarga de página.
    • El caché de estados tiene invalidación por evento (TTL máximo configurable, defecto 5 min).
    • Si el gestor no responde, se mantiene el último estado conocido y se muestra un indicador de "última actualización hace X min".
  4. Fallback por datos incompletos
    • Si un servicio carece de categoría o estado, se asigna automáticamente Sin programación actual y se registra una alerta interna.
    • Los servicios sin datos completos no se ocultan del catálogo; se muestran con etiqueta de fallback.
  5. Filtrado y rendimiento
    • El catálogo ofrece filtro por estado de disponibilidad aplicable en tiempo real.
    • La carga inicial del catálogo con etiquetas debe completarse en ≤ 2 s (p95) para listados de hasta 200 servicios.
    • Las etiquetas son accesibles (contraste WCAG AA mínimo) y compatibles con lectores de pantalla.
2. Tecnología Stack propuesto
Catálogo de Servicios (fuente)
  • Campo status con enum estricto: available | upcoming | closed | unscheduled.
  • Campo category con jerarquía: area_id + subarea_id.
  • Maestro versionado en base de datos relacional o Delta Lake.
Integración y Eventos
  • Webhook del gestor de servicios para propagación de cambios de estado.
  • Caché con invalidación por evento (Redis o similar).
  • Capa de Motor / API que expone el catálogo enriquecido con etiquetas.
Frontend / UI
  • Design tokens para colores de estado (sin hardcoding de colores en componentes).
  • Componente de tarjeta reutilizable con slot de etiquetas de categoría y estado.
  • Filtro por estado integrado en el listado del catálogo.
Observabilidad y Auditoría
  • Log de cada cambio de clasificación: campo, valor anterior, valor nuevo, autor, timestamp.
  • Alerta interna cuando se detectan servicios sin categoría o sin estado válido.
  • Dashboard interno de salud del catálogo (% servicios sin etiqueta, última sincronización).
3. Flujo Técnico y de Datos
ACTORES
  • Usuario / Actor: consulta el recomendador.
  • CRM / Dynamics: provee etiquetas visibles por servicio.
  • Catálogo Servicios: aplica reglas de clasificación.
  • Motor / API: actualiza frescura de disponibilidad.
  • Frontend / UI: gestiona fallback por datos incompletos.
SECUENCIA PRINCIPAL
  1. CRM / Dynamics entrega etiquetas visibles por servicio (categoría + estado).
  2. Catálogo Servicios aplica reglas de clasificación consistentes y trazables al Motor / API.
  3. Motor / API envía actualización y frescura de la disponibilidad al Frontend / UI.
  4. Frontend / UI devuelve fallback por datos incompletos al Motor / API.
  5. Motor / API devuelve confirmación de rendimiento al Usuario.
SALIDA
  • Tarjeta de servicio con etiqueta de categoría (área > subárea).
  • Badge de estado con color diferenciado.
  • Contador de cupos cuando aplica.
  • Indicador de última actualización.
Pasos del Pipeline de Clasificación
  1. Ingesta: el gestor de servicios publica un evento de cambio de estado o categoría.
  2. Validación: el Motor / API valida el enum de estado y la jerarquía de categoría; rechaza valores inválidos.
  3. Enriquecimiento: se adjuntan metadatos de cupos, fechas de vigencia y tags estratégicos.
  4. Caché: el resultado enriquecido se guarda en caché con TTL; se invalida al recibir nuevo webhook.
  5. Fallback: si faltan datos, se asigna unscheduled y se registra alerta.
  6. Exposición: el Frontend / UI consulta el catálogo y renderiza tarjetas con etiquetas.

4. Diseño — Etiquetas y Estados Visibles
Estados de Disponibilidad y Colores
  • Disponible — verde: inscripciones abiertas.
  • Próximamente — naranja: fecha conocida futura.
  • Sin programación — gris: sin fecha definida.
  • Cerrado — rojo: fuera de vigencia.
Elementos de la Tarjeta de Servicio
  • Nombre del servicio (título principal).
  • Breadcrumb de categoría: Área > Subárea.
  • Badge de estado con color semántico.
  • Cupos disponibles (cuando status = available).
  • Indicador de "Última actualización: hace X min".
  • Botón de acción contextual (inscribirse / avisarme / ver detalle).
5. Casos de Prueba Mínimos
# Escenario Resultado Esperado
1 Etiquetas visibles en tarjeta Cada tarjeta muestra categoría (área > subárea) y badge de estado con color correcto según design token.
2 Enum estricto de estado Un estado con valor "activo" (fuera del enum) es rechazado con error 400 y se registra alerta.
3 Actualización en tiempo real Tras webhook de cambio de estado a available, la tarjeta actualiza el badge sin recarga en ≤ 5 s.
4 Fallback por datos incompletos Servicio sin categoría aparece con etiqueta Sin programación y se genera alerta interna.
5 Filtro por estado Al seleccionar filtro "Disponible", solo se muestran servicios con status = available.
6 Rendimiento (p95) Listado de 200 servicios con etiquetas carga en ≤ 2 s (p95) medido en entorno de prueba.
6. Glosario
Enum estricto
Conjunto cerrado de valores permitidos para un campo; cualquier valor externo es rechazado.
Design token
Variable de diseño (color, tipografía, espaciado) centralizada que se aplica a todos los componentes.
Webhook
Notificación HTTP enviada por el gestor de servicios cuando ocurre un cambio de estado.
Caché con TTL
Almacenamiento temporal de respuestas con expiración automática por tiempo o por evento.
Fallback
Valor o comportamiento por defecto aplicado cuando los datos esperados están incompletos o ausentes.
Cupos disponibles
Número de plazas activas restantes para inscripción a un servicio con capacidad limitada.
Nota de alcance: Esta HU cubre exclusivamente la clasificación y presentación visual de servicios con etiquetas de categoría y estado. La lógica de suscripción a notificaciones de disponibilidad futura se gestiona en HU 2.13.2.
